## TL;DR

- **H1 (Contextual prefix drift) — опровергнута.** Префиксы mn10/dn22 буквально содержат «satipaṭṭhāna» и «Satipaṭṭhāna Sutta».
- **H4 (per-channel split) — confirmed.** На исходном запросе `dense=— (None), sparse=48/74` для mn10/dn22 в `dharma_v2` — dense embedding **вообще не выдаёт их в top-200**, sparse-канал спасает.
- **H5 (query length / specificity) — confirmed.** Короткий definitional запрос ловит «commentary-in-disguise» сутт-самьютты sn47.x/sn52.x; expansion до полного gloss («What are the four foundations of mindfulness?») мгновенно поднимает mn10 на **#1**.
- **H3 (foundational-vs-derivative bias) — confirmed.** sn47.x — серия эпизодов где термин повторяется буквально в каждом — embedding-similarity завышенная. mn10/dn22 содержат термин в title и узловых местах, остальной текст — конкретные практики (breathing, postures, repulsive contemplation).
- **H2 (chunk-level pollution) — нет.** Чанки mn10:0.1 / dn22:0.1 — нормальные opening sections.
- **Multi-causal:** реальный root cause = **H4 + H5 + H3 одновременно**. Embedding бьёт mn10/dn22 в dense-канале только на **расширенных** definitional queries.
- **Generalisation:** аналогичный паттерн на `sn56.11` (dukkha) и `sn22.59` (anatta) — оба отсутствуют в top-200 на `dharma_v2`. Right View (`mn117`) — **исключение**, т.к. термин буквально в title; BM25 ловит точное совпадение.

Это **systemic** bug, не isolated. Затрагивает foundational сутты с **одним коротким** definitional-запросом. **Не затрагивает** сутты где термин уже в title (mn117 «Right View»).

## Root cause analysis

### Что происходит

1. **Запрос «What is satipaṭṭhāna?»** — 4 токена, низкая семантическая плотность.
2. **BGE-M3 dense embedding** на короткий запрос усредняет к **topic-cluster centroid** — место в семантическом пространстве, окружённое всеми чанками о satipaṭṭhāna.
3. **Sutta-saṃyutta sn47.x** — десятки эпизодов, каждый чанк содержит термин 5-10 раз → их dense-вектора **сильнее** прижаты к topic-centroid (более «narrow» в этой теме).
4. **Foundational mn10/dn22** — opening + body practice + breathing + repulsive + elements + ... — большая семантическая дисперсия. Title-сегмент `mn10:0.1` тематичен, но остальные сегменты дрейфуют в сторону meditation techniques. Усреднённый sentence-level embedding не фокусирует.
5. **Sparse-канал** (lexical TF-IDF-like) — тоже даёт sn47.x приоритет, потому что в них **частота вхождения термина** выше per chunk.
6. **BM25-канал на en-query «satipaṭṭhāna» с диакритикой** — `text_ascii_fold` нормализует диакритику, но запрос идёт через FTS `simple` config; диакритика остаётся как есть → match-rate 0 (видим `bm25=—` в каждой строке Phase A1/A2).
7. **Без BM25 в первой пятёрке** — fusion отдаёт приоритет dense+sparse, оба тянут к sn47.x.

### Multi-causal nature

Это **не одна** проблема. Минимум **три** причины складываются:

- **H4** — dense channel mute on short query → no balanced signal
- **H5** — query specificity insufficient → topic-centroid match wins
- **H3** — foundational suttas dispersed in semantic space → derivative repetitive content wins by surface frequency

И ещё:

- **BM25 не нормализует диакритику** (`bm25=—` везде на A1)
- **No glossary entry** для «satipaṭṭhāna» (Pali glossary rag-day-23 expand_pali в проде по умолчанию `True` — но видимо не работает на этом термине, надо проверить)

## Recommendations (priority по ROI)

### 1. Definitional query expansion (HIGH priority, cheap)

Перед encoding короткого `What is X?` запроса — расширять через template:

```
"What is {term}?" → "What is {term}? Discourse on {term}. Definition of {term}. Foundations of {term}."
```

**A3 показал:** `What are the four foundations of mindfulness?` мгновенно поднимает mn10 → #1. Это **детерминистический** rewrite на pattern «What is X?», не требует ML. Можно добавить в `src/processing/glossary.py` рядом с existing `expand_query` — отдельная стратегия для definitional patterns.

**Ожидаемый эффект:** ref_hit@5 на foundational definitional queries (~10% golden) → +~3-5 pp.

### 2. Pali / EN glossary lookup для term-only queries (HIGH priority, cheap)

Если query содержит **только** один Pali термин (после стоп-слов) — посмотреть в Pali glossary для **canonical sutta references** и буст.

Текущий `Glossary` (concept 14) расширяет термин синонимами; тут нужна **обратная** связь — термин → foundational сутры. Для этого нужен curated mapping (small): `{satipaṭṭhāna: [mn10, dn22], dukkha: [sn56.11, sn22.84], anatta: [sn22.59, mn22], ...}` — 30-50 пар на старте.

**Где живёт:** новый `data/glossary/foundational.yaml`. Lookup query → boost result rank if matching expected work in retrieved set.

### 3. BM25 диакритика fix (MEDIUM, technical)

Сейчас BM25 на en-query «satipaṭṭhāna» промахивается из-за диакритики. Нужно либо нормализовать query перед FTS-запросом (apply same ASCII-fold что применяется к корпусу), либо использовать `unaccent` extension в Postgres FTS config.

**Ожидаемый эффект:** `bm25=—` → `bm25=N` для term-with-diacritics queries; даст third channel signal для balance.

### 4. Foundational-sutta metadata + boost (MEDIUM, requires curation)

Добавить в `Work` модель flag `is_foundational: bool` (или score `foundational_weight: float`). Curator-список 30-50 ключевых сутт. Boost factor применяется в RRF fusion stage:

```python
rrf_score *= 1.5 if work.is_foundational else 1.0
```

**Trade-off:** требует buddhologist judgment на curation list. Хорошая задача для B-001.

### 5. Title-only named vector (LOW, expensive — separate ingest)

Отдельный Qdrant named vector `bge_m3_title` где embed только `Work.title + Work.title_pali`. На definitional queries этот канал даёт strong signal.

Cost: re-embed всех works × small text → cheap. Но добавляет complexity (4-й канал в hybrid pipeline). Ставлю **low** до тех пор, пока (1)+(2)+(3) не выложены и не измерены.

### Не делаем (из rejected hypotheses)

- **H1 — Contextual Retrieval prompt fix** — не нужно. Префиксы корректные.
- **H2 — chunk pollution** — тоже корректные. Не fix-target.
