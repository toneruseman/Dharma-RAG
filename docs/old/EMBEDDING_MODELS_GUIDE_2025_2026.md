# Полное руководство по моделям текстовых эмбеддингов для RAG и семантического поиска в 2025–2026

**Рынок моделей эмбеддингов достиг точки конвергенции: разница между лучшими моделями составляет менее 3% по бенчмаркам, и выбор конкретной модели менее важен, чем качество чанкинга, реранкинга и пайплайна оценки.** Тем не менее, выбор модели определяет стоимость, задержку, мультиязычность и возможности гибридного поиска — и здесь различия фундаментальны. В этом отчёте систематизированы все основные варианты с конкретными цифрами, бенчмарками и практическими рекомендациями. Особое внимание уделено поддержке русского языка, где GigaEmbeddings и BGE-M3 лидируют среди открытых моделей, а Qwen3-Embedding-8B устанавливает новый стандарт мультиязычности.

## Коммерческие API: цена, качество и подводные камни

### Google Gemini Embedding

**Gemini Embedding 001** — лидер английского MTEB с результатом **68.32** и лучшим retrieval-скором **67.71 nDCG@10** среди всех коммерческих моделей. Размерность 3072 (с поддержкой Matryoshka — сокращение до 768 или 1536), 100+ языков, 8 типов задач (RETRIEVAL_QUERY, RETRIEVAL_DOCUMENT, CLASSIFICATION и др.). Цена — $0.15/MTok, есть бесплатный тариф (1500 запросов/день). Основной недостаток — ограничение контекста до **2048 токенов** через Gemini API (до 8192 через Vertex AI), что критично для длинных документов.

**Gemini Embedding 2** (март 2026, превью) — первая мультимодальная модель эмбеддингов, объединяющая текст, изображения, видео и аудио в одном 3072-мерном пространстве. Кросс-языковой retrieval достигает **0.997** — фактически идеальный результат. Цена выше: $0.20/MTok (текст), $0.45 (изображения), $6.50 (аудио). Недостаток: привязка к экосистеме Google, нет возможности self-hosting.

### Voyage AI (серия Voyage 4)

Voyage AI, приобретённая MongoDB за $220M в феврале 2025, выпустила серию **Voyage 4** в январе 2026 — первые embedding-модели на архитектуре **Mixture-of-Experts (MoE)**, что снижает стоимость обслуживания на 40%. Ключевое нововведение — **единое векторное пространство** для всех моделей серии: можно индексировать документы через voyage-4-large, а искать через voyage-4-lite без потери совместимости.

| Модель | Цена/MTok | Размерность | Контекст | Примечание |
|--------|-----------|-------------|----------|------------|
| voyage-4-large | $0.12 | 256–2048 | 32K | Лучший retrieval NDCG@10 |
| voyage-4 | $0.06 | 256–2048 | 32K | Баланс цена/качество |
| voyage-4-lite | $0.02 | 256–2048 | 32K | Как OpenAI 3-small по цене, 4× контекст |

По заявлению Voyage AI, voyage-4-large превосходит Gemini Embedding 001 на **3.87%**, Cohere v4 на **8.2%** и OpenAI text-embedding-3-large на **14.05%** по NDCG@10. Эти цифры — самоотчёт вендора, независимая верификация ограничена. Сообщество Reddit отмечает: «Voyage всегда лучший, но с высокой задержкой». Бесплатно — **200M токенов** на каждую модель серии.

### OpenAI text-embedding-3

**Не обновлялись более двух лет** (с января 2024). text-embedding-3-small ($0.02/MTok, 1536 dims, MTEB ~62.3) остаётся самым популярным по инерции — глубокая интеграция с экосистемой OpenAI. text-embedding-3-large ($0.13/MTok, 3072 dims, MTEB ~64.6) поддерживает Matryoshka, но значительно уступает Gemini и Voyage по качеству. Не поддерживают instruction-префиксы и типы задач — существенный недостаток.

Критический минус: **ada-002 по-прежнему не deprecated**, но стоит $0.10/MTok при MTEB 61.0 — в 5 раз дороже text-embedding-3-small при худшем качестве. Если вы всё ещё используете ada-002, миграция на 3-small — мгновенная экономия при улучшении качества.

### Cohere Embed v4

Революционный контекст в **128 000 токенов** — самый длинный среди всех embedding-моделей, что делает его идеальным для работы с целыми документами без чанкинга. Мультимодальность (текст + изображения + PDF), 100+ языков, Matryoshka (256–1536), поддержка бинарной/байтовой квантизации с **83% экономией хранения**. Цена — $0.12/MTok (текст).

Однако предыдущая версия **embed-v3 имела контекст всего 512 токенов** — радикальное ограничение, требовавшее агрессивного чанкинга. Переход на v4 устранил эту проблему, но AIMultiple обнаружил, что Cohere v4 может быть «ловушкой релевантности»: находит семантически похожие, но не обязательно корректные документы.

### Jina AI Embeddings

**Jina v3** (570M параметров, 1024 dims, 8192 токенов) — единственная коммерческая модель с **task-specific LoRA-адаптерами** (retrieval.query, retrieval.passage, separation, classification, text-matching). Поддерживает 89 языков, включая русский. Цена: пакетная система токенов, 10M бесплатных токенов. Лицензия: CC-BY-NC-4.0 для весов, API для коммерческого использования.

**Jina v4** (3.8B параметров, Qwen2.5-VL-3B, мультимодальная) — поддержка ColBERT-стиля late interaction + dense retrieval. SOTA на ViDoRe (визуально насыщенные документы). Проблема лицензии: изначально выпущена под cc-by-nc-4.0 «по ошибке», фактическая лицензия — Qwen Research License.

### Mistral Embed

Два продукта: **mistral-embed** ($0.10/MTok, 1024 dims, 8192 токенов) — посредственное качество, фиксированная размерность, без Matryoshka. **Codestral Embed 2505** ($0.15/MTok, 1536 dims, 32K токенов) — специализированная модель для кода. Важно: ранние отчёты указывали цену $0.01/MTok для mistral-embed — это было ошибкой, реальная цена $0.10/MTok.

### Amazon Titan Embeddings V2

$0.02/MTok, 1024 dims (настраивается: 256/512/1024), 8192 токенов, 100+ языков. Не публикует MTEB-скоры. Главный недостаток: **доступен исключительно через AWS Bedrock** — полная привязка к экосистеме AWS. Кросс-языковой поиск субоптимален.

## Открытые модели: от компактных до мощных

### BGE-M3 — уникальный гибрид (BAAI, MIT)

**Единственная модель, поддерживающая все три режима retrieval одновременно**: dense (1024-мерные векторы), sparse (обученные лексические веса, аналог SPLADE/BM25) и multi-vector/ColBERT (поэтому эмбеддинги). Формула гибридного ранжирования: `s = w1·s_dense + w2·s_sparse + w3·s_colbert`.

568M параметров, XLM-RoBERTa backbone, 8192 токенов, 100+ языков. VRAM: **~1.1 ГБ в FP16** — помещается на любую современную GPU. Лицензия MIT. На ruMTEB набирает **60.8** среди non-instruct моделей — один из лучших результатов для retrieval на русском. Улучшенная версия **deepvk/USER-bge-m3** достигает 0.799 на Encodechka.

Недостатки: данные обучения смещены в сторону high-resource языков; ColBERT-векторы требуют значительного хранилища для больших корпусов (>1 ТБ для средних коллекций); sparse vocabulary ограничен токенизатором XLM-RoBERTa.

### Qwen3-Embedding — новый стандарт мультиязычности (Alibaba, Apache 2.0)

Семейство 2025 года с вариантами **0.6B, 4B, 8B** параметров. Модель 8B заняла **#1 на MTEB Multilingual** с результатом **70.58** и #1 на MTEB-Code. Гибкая размерность от 32 до 7168 (Matryoshka), контекст 32K–128K токенов, 100+ языков включая языки программирования. Полностью открытая лицензия Apache 2.0.

| Вариант | Параметры | VRAM (FP16) | VRAM (INT4) | MTEB ML |
|---------|-----------|-------------|-------------|---------|
| Qwen3-Embedding-0.6B | 600M | ~600 МБ | ~300 МБ | Хороший |
| Qwen3-Embedding-4B | 4B | ~8 ГБ | ~2 ГБ | Высокий |
| Qwen3-Embedding-8B | 8B | ~16 ГБ | ~5 ГБ (Q4) | **70.58** |

Модель 8B помещается на RTX 4090 с квантизацией Q4_K_M (~5 ГБ VRAM). Instruction-aware: инструкция к задаче улучшает качество на 1–5%.

### NV-Embed-v2 (NVIDIA, CC-BY-NC-4.0)

Рекордсмен английского MTEB v1 с **72.31** (62.65 на retrieval). Архитектура: Mistral-7B с bidirectional attention и Latent-Attention pooling. Однако **лицензия CC-BY-NC-4.0 запрещает коммерческое использование**, VRAM ~30–46 ГБ для инференса, скорость ~200 мс/запрос. Для исследований — отличный бенчмарк; для продакшена — нежизнеспособен из-за лицензии и ресурсоёмкости.

### Microsoft Harrier-OSS-v1 (март 2026, MIT)

Преемник E5 от Microsoft. Три варианта: 270M (Gemma 3), 0.6B (Qwen 3), 27B (Gemma 3). Модель 27B достигает **MTEB v2: 74.3** — SOTA на мультиязычном бенчмарке v2. MIT-лицензия, 32K контекст, 100+ языков. Требует ~54 ГБ VRAM в BF16 для 27B — практически непригодна для локального развёртывания без серверных GPU.

### Nomic Embed Text v2 MoE (февраль 2025, Apache 2.0)

Первая MoE-архитектура для embedding-модели: 475M параметров, но только **305M активных** при инференсе. SOTA для класса ~300M на BEIR (английский retrieval) и MIRACL (мультиязычный). ~100 языков, Matryoshka (768→256). Однако максимальный контекст — **512 токенов** (значительный регресс от 8K у v1.5). Apache 2.0 с полной открытостью (веса, код, данные обучения).

### Прочие значимые модели

**mxbai-embed-large-v1** (Mixedbread AI, 335M, Apache 2.0): MTEB 64.68 — превосходит OpenAI text-embedding-3-large при 335M параметрах. Matryoshka до 64 dims, бинарная квантизация сохраняет 91.8% качества. Недостаток: 512 токенов, только английский.

**stella_en_1.5B_v5** (MIT, 1.5B): Настраиваемая размерность 256–8192, дуальная система промптов (s2p_query для retrieval, s2s_query для similarity). ~6.2 ГБ VRAM. Конкурирует с 7B-моделями при значительно меньших ресурсах.

**Snowflake Arctic Embed 2.0** (декабрь 2024, Apache 2.0, 568M): Мультиязычность без потери качества на английском — уникальное достижение. MRL + quantization-aware training: 128 байт/вектор (96× сжатие по сравнению с OpenAI). >100 документов/сек на A10 GPU, <10 мс задержки.

**EmbeddingGemma-300M** (Google DeepMind, 2025): Оптимизирована для edge/on-device — <200 МБ RAM с квантизацией, <22 мс на EdgeTPU. 100+ языков. Идеальна для мобильных приложений.

**all-MiniLM-L6-v2** (22M параметров, 384 dims): Устаревшая, но по-прежнему самая используемая модель для прототипов. MTEB ~56, 256 токенов контекста, ~50 МБ — работает на любом CPU за 16 мс.

## Русский язык: специализированные модели и бенчмарки

### GigaEmbeddings — лидер ruMTEB

**GigaEmbeddings** (ai-sage/Giga-Embeddings-instruct) — SOTA на ruMTEB с результатом **69.1**. Построена на GigaChat-3B (decoder-only), оптимизирована для русского и английского. 25% transformer-слоёв обрезано для эффективности. Открыта на HuggingFace. Для задач, требующих глубокого понимания русского текста (юридические документы, корпоративный поиск, образование), это лучший выбор на апрель 2026.

### ruMTEB и RusBEIR — бенчмарки для русского

**ruMTEB** (NAACL 2025): 23 датасета по 7 категориям задач — Classification (9), Clustering (3), Retrieval (3), STS (3) и другие. Интегрирован в официальный фреймворк MTEB. Результаты instruct-моделей:

| Модель | ruMTEB avg | Тип |
|--------|-----------|-----|
| **GigaEmbeddings** | **69.1** | Instruct |
| E5-mistral-7b-instruct | 64.9 | Instruct |
| mE5-large-instruct | 64.7 | Instruct |
| BGE-M3 | 60.8 | Non-instruct |
| mE5-large | 60.4 | Non-instruct |
| ru-en-RoSBERTa | 60.4 | Non-instruct |

**RusBEIR** (апрель 2025): 17 датасетов для zero-shot IR на русском. Ключевой вывод: **BM25 остаётся сильным baseline** для русского языка, а BGE-M3 и mE5-large показывают лучшие нейронные результаты. Морфологическая предобработка критична для морфологически богатого русского.

Между instruct и non-instruct моделями наблюдается устойчивый разрыв в ~4 пункта. Для retrieval-задач на русском BGE-M3 лидирует среди non-instruct моделей, но instruct-модели (E5-mistral-7B, mE5-large-instruct) существенно превосходят.

## Matryoshka, гибридный поиск и ColBERT

### Matryoshka Representation Learning захватил рынок

Практически все новые модели 2025–2026 поддерживают MRL — возможность усекать эмбеддинги до меньшей размерности с контролируемой потерей качества. Общее правило: **уменьшение размерности вдвое сохраняет качество практически без потерь**, до 1/3 — с минимальными потерями. OpenAI text-embedding-3-large на 256 измерениях превосходит старую ada-002 на полных 1536 — то есть 6× меньший вектор при лучшем качестве.

Для Qwen3-Embedding исследование на Medium показало sweet spot около **400 измерений**: выше 650 прирост пренебрежимо мал. Практическая стратегия: использовать MRL-префикс (256 dims) для грубой фильтрации 100–200 кандидатов, затем полные эмбеддинги для финального ранжирования — **50–80% сокращение задержки** с минимальной потерей точности.

Комбинация MRL (128 dims) + скалярная квантизация даёт **77.9% сокращение индекса** (с 172 МБ до 38 МБ для 100K векторов). Важно: после ручного усечения размерности необходимо перенормализовать векторы — некоторые провайдеры (Voyage) делают это автоматически, Google — нет.

### BGE-M3 — единственный полноценный гибрид

BGE-M3 остаётся единственной моделью, генерирующей три типа векторов одновременно: dense (семантические 1024-мерные), sparse (обученные лексические веса для точного совпадения ключевых слов) и multi-vector (ColBERT-стиль для fine-grained matching). Интегрирован с Milvus и Vespa.

Dense-only поиск упускает точные совпадения (артикулы, редкие имена), sparse-only — семантическое понимание. Гибрид dense+sparse+ColBERT rerank даёт лучшие результаты во всех бенчмарках. Практическая рекомендация: dense+sparse для retrieval, ColBERT как реранкер (экономичнее, чем ColBERT для primary retrieval, где хранение per-token эмбеддингов требует >1 ТБ для средних корпусов).

- **8 ГБ VRAM** (RTX 4060): all-MiniLM-L6-v2, nomic-embed-text-v1.5, bge-base. НЕ помещается: BGE-M3, любые 7B.
- **16 ГБ VRAM** (RTX 4080): всё выше + BGE-M3 (~1.1 ГБ FP16), mE5-large, Qwen3-Embedding-0.6B. НЕ помещается: 7B-модели в FP16.
- **24 ГБ VRAM** (RTX 4090): всё выше + 7B-модели с INT4/INT8 квантизацией (E5-Mistral: ~3.5–7 ГБ INT4/INT8), Qwen3-Embedding-8B с Q4_K_M (~5 ГБ).

### API vs Self-hosting: точка безубыточности

| Метод | Стоимость/MTok | Примечание |
|-------|---------------|------------|
| Self-hosted BGE-M3 на RTX 4090 (RunPod) | **~$0.002–0.005** | $0.39/ч, высокая пропускная способность |
| Self-hosted на AWS g5.xlarge | **~$0.005–0.01** | $1.01/ч |
| Google text-embedding-005 | $0.006 | Самый дешёвый API |
| OpenAI text-embedding-3-small | $0.02 | Популярный выбор |
| Cohere embed-v4 | $0.12 | Мультимодальный |

**Self-hosted BGE-M3 на AWS g5.xlarge (~$730/мес)** выгоднее OpenAI 3-small при объёме >36.5 млрд токенов/месяц. Реальный кейс: компания перешла с OpenAI 3-large ($7800/мес за 2M daily embeddings) на self-hosted BGE-M3 ($730/мес) — **экономия $85000+ в год**.

Однако для дешёвых API (Google $0.006/MTok) self-hosting окупается только при гигантских объёмах (100B+ токенов/мес). Скрытые затраты self-hosting: 1.5–2 FTE DevOps ($270K–$550K/год), инфраструктурный множитель 2.5–3× на GPU-стоимость, egress-трафик ($87–120/ТБ у гиперскейлеров).

**Правило принятия решения**: <1B токенов/мес → API; 1–10B → API или гибрид в зависимости от требований к приватности; >10B → self-hosting начинает экономить; при требованиях к приватности данных → self-hosting вне зависимости от объёма.

## Критический анализ: что бенчмарки скрывают

### Самоотчётные результаты MTEB

MTEB-скоры публикуются самими авторами моделей — **независимая верификация отсутствует**. MTEB v1 и v2 несовместимы (модель с 72.31 на v1 и модель с 74.3 на v2 нельзя сравнивать напрямую). В ответ на это Hugging Face запустил **RTEB** (октябрь 2025) с приватными датасетами — модели, блестящие на публичных бенчмарках, «спотыкаются на невиданных данных RTEB».

### Переобучение на бенчмарки

Авторы MTEB документировали: «модели с высшими рангами достигают своих скоров обучением на benchmark-задачах, хотя модели с более низкими скорами могут лучше обобщаться». **~50% retrieval-датасетов MTEB происходят из QA-задач**, потенциально отдавая предпочтение лексическому совпадению перед глубоким семантическим пониманием.

Legal RAG Bench (октябрь 2025) обнаружил, что «производительность на существующих legal retrieval бенчмарках не коррелировала» с реальным качеством из-за data leakage и плохого качества разметки (датасет AILA, составляющий 25% legal-сегмента MTEB, содержал «полностью нерелевантные» пары query-passage).

### Конвергенция качества

Пользователь HuggingFace Forums (ноябрь 2025) протестировал 13 моделей на 8 датасетах: **~85% моделей находятся в пределах 50 ELO-пунктов**, топ-4 разделены лишь ~23.5 ELO. Вывод: «выбор идеальной embedding-модели — уже не критическое решение». Гораздо важнее: качество чанкинга, использование реранкинга (даёт больший прирост, чем смена модели) и **fine-tuning на доменных данных** (+10–30% для специализированных задач).

### Риски vendor lock-in и deprecation

Реальный инцидент: «Нашу embedding-модель deprecated за ночь. Каждый RAG-запрос начал возвращать 404» (DEV Community, миграция с text-embedding-004 на text-embedding-3-large). Cohere deprecated несколько моделей к 4 апреля 2026. Google deprecated text-embedding-004 к 14 января 2026.

Смена модели = пере-эмбеддинг всего корпуса. Практический совет: прогоните новую модель на 1–5% выборки; если NDCG@10 улучшается менее чем на 3–5%, миграция скорее всего не стоит затрат. Самая безопасная стратегия — модели с открытым кодом (Qwen3 Apache 2.0, BGE-M3 MIT, Arctic Embed Apache 2.0).

## Практические рекомендации по сценариям

### Лучшее качество без ограничений бюджета

**Voyage 4 Large** ($0.12/MTok) для retrieval + **Cohere Embed v4** для длинных документов (128K контекст). Для self-hosted: **Qwen3-Embedding-8B** (Apache 2.0) на H100 или **Harrier-OSS-v1-27B** (MIT) для абсолютного максимума.

### Лучший баланс цена/качество

**BGE-M3** (MIT, self-hosted на RTX 4090 за $0.39/ч) — гибридный поиск, 100+ языков, 8K контекст, ~1.1 ГБ FP16. Для API: **voyage-4-lite** или **OpenAI text-embedding-3-small** ($0.02/MTok). Для бóльшего качества: **Qwen3-Embedding-4B** — баланс между 0.6B и 8B (8 ГБ VRAM в FP16).

### Минимальный бюджет / бесплатные варианты

- **Бесплатные API**: Google Gemini (1500 запросов/день), Voyage AI (200M токенов), Jina (10M токенов)
- **Self-hosted**: all-MiniLM-L6-v2 (CPU, 50 МБ), nomic-embed-text-v1.5 (CPU/GPU, 274 МБ), BGE-M3 (GPU, 1.1 ГБ)
- **Стратегия**: Начать с all-MiniLM-L6-v2 для прототипа → BGE-M3 для продакшена

### Специально для русского языка

1. **GigaEmbeddings** — безусловный лидер ruMTEB (69.1), специально обучена на русском
2. **E5-mistral-7b-instruct** — второе место ruMTEB (64.9), но требует 14 ГБ VRAM
3. **mE5-large-instruct** — компактная альтернатива (64.7, ~1 ГБ FP16)
4. **BGE-M3** — лучший non-instruct вариант для русского (60.8), плюс гибридный поиск
5. **Qwen3-Embedding-8B** — сильная мультиязычность, но ruMTEB-результаты не опубликованы

Для максимального качества на русском при ограниченных ресурсах: **BGE-M3** (dense+sparse гибрид) + CrossEncoder реранкер. Для максимального качества без ограничений: **GigaEmbeddings** или **E5-mistral-7b-instruct**.

### Для кода

**Qwen3-Embedding-8B** (#1 MTEB-Code), **Gemini Embedding 2** (MTEB Code: 84.0), **Voyage Code-3** ($0.18/MTok).

### Для мультимодальности (текст + изображения + PDF)

**Gemini Embedding 2** (все модальности в одном пространстве), **Cohere Embed v4** (текст+изображения+PDF, 128K контекст), **Jina v4** (SOTA на ViDoRe для визуально насыщенных документов).

## Заключение: что действительно важно в 2026 году

Экосистема embedding-моделей прошла фазу экспоненциального роста и достигла плато качества. Три инсайта критически важны при принятии решения:

**Первый**: разница между топ-10 моделями по бенчмаркам составляет менее 3%, но разница в стоимости — до 100×. Self-hosted BGE-M3 стоит ~$0.002/MTok, а Voyage 3-large — $0.18/MTok. При эквивалентном качестве для 90% задач, выбор определяется экономикой, приватностью и требованиями к контексту — не бенчмарками.

**Второй**: для русского языка ландшафт радикально изменился с появлением ruMTEB и GigaEmbeddings. Впервые существует специализированная модель, значительно превосходящая универсальные мультиязычные решения на русском (69.1 vs 60.8 для BGE-M3). Fine-tuning на доменных русскоязычных данных может дать +10–30% прироста — инвестиция, часто более ценная, чем смена модели.

**Третий**: гибридный поиск (dense+sparse) перестал быть экзотикой и стал стандартом. BGE-M3 остаётся единственной моделью, нативно поддерживающей все три режима, что делает её уникально ценной для production-систем, где критична полнота поиска (recall). Вместо погони за «лучшей» моделью постройте evaluation pipeline на своих данных — это окупится значительно больше, чем любая миграция между топовыми моделями.
