# Dharma-RAG 2026: архитектура, оценка и fine-tuning

Три стратегических решения определят успех Dharma-RAG в ближайшие 12 месяцев. **Во-первых, оставайтесь на Qdrant в Phase 1–2 и откладывайте миграцию на pgvector/pgvectorscale до Phase 3** — миф «одна коллекция = одна модель» опровергнут named vectors с Qdrant 1.10, а binary quantization 1.5-bit (v1.15) плюс tiered multi-tenancy (v1.16) закрывают все реальные ограничения. **Во-вторых, граф — это не ядро RAG, а опциональный слой:** Contextual Retrieval от Anthropic (+35% precision) и hybrid BGE-M3 (+49%) дают больший ROI, чем любой knowledge graph на корпусе \<1M чанков; формальный граф (Apache AGE в том же Postgres) добавляется только в Phase 2 и только если ≥3-hop запросы станут заметной долей трафика. **В-третьих, fine-tuning — последний шаг, не первый:** для RAG качество дистрибутируется по retrieval; FT BGE-M3 и reranker даст +5–15 pp Hit@5 за €1 электричества, тогда как FT генеративной LLM — дорогой эксперимент с marginal gain.

Эти три вывода идут против общих настроений AI-community 2025–2026, где «GraphRAG everywhere» и «agentic fine-tuning» стали модой. Для узкого буддийского корпуса с уже явной структурой (nikāya, PTS-цитации, SuttaCentral parallels.json) LLM-extracted графы приносят шум больше, чем сигнала, а правила — лучше LLM. Budget $4625/12mo и hardware CX32→CCX33→локальный 2×48 GB VRAM реалистично закрывают все Phase 1–3 задачи без cloud-зависимости на критическом пути.

## Qdrant остаётся оптимумом: развенчание четырёх мифов

**Миф о named vectors опровергнут окончательно с версий 1.10–1.16.** В одну Qdrant-коллекцию кладутся четыре пространства одной точки: `dense_bgem3` (1024d fp16/int8), `dense_v2` (slot для будущей миграции, например multilingual-e5-large или Jina v3), `sparse_bgem3` c IDF-модификатором (v1.10+), и `colbert` как multivector с MAX_SIM comparator и отключённым HNSW (`m=0`), используемый только для rerank-stage через Query API. Payload при этом **не дублируется** — одна запись метаданных на все четыре вектора. Это архитектурное отличие от «двух коллекций» или «PG + Qdrant sidecar»: миграция между эмбеддинг-моделями реализуется паттерном **dual-write + backfill + alias swap** без downtime, с conditional updates (v1.16) для защиты от race conditions.

**Практический рецепт квантизации для Phase 2 (900K × 1024d):** int8 scalar с `always_ram=true` и оригиналами на диске даёт 99%+ recall при ~1.4 GB RAM на dense-индекс; если RAM критичен, Binary Quantization 1.5-bit (v1.15) с `oversampling=3.0` и FP32 rescore топ-200 режет до ~0.25 GB ценой 1–2 pp recall, что полностью перекрывается ColBERT-rerank. **BQ 1-bit на BGE-M3 1024d — пограничный случай:** распределение ненормальное на Pali/Sanskrit, поэтому 1-bit обычно теряет 5+ pp recall; именно 1.5-bit — sweet spot для dim 768–1024. ColBERT fp16 on-disk (~38 GB на 900K × 180 токенов × 128d) никогда не резидентен целиком — page cache и rerank top-200 из dense+sparse prefetch держат p95 под 150 ms на CCX33. **Milvus 2.6 с IVF_RABITQ** теоретически даёт 32× компрессию с recall ≥94%, но CCX33 (AMD EPYC Milan) лишён AVX-512 VPOPCNTDQ, так что заявленные цифры недостижимы; на самостоятельном 7950X3D ситуация та же.

**pgvector 0.8 + pgvectorscale StreamingDiskANN — реальный конкурент только на Phase 3.** Timescale-бенчмарк показывает 471 QPS при 99% recall на 50M × 768d против 41 QPS у Qdrant (11.4× преимущество, но на конкретном Cohere-датасете и вендорском сетапе). **Критическая слабость для Dharma-RAG:** нативный multivector late-interaction отсутствует — ColBERT придётся хранить как `vector[]` и считать MAX_SIM в SQL/UDF, что на 900K × 180 токенов убьёт latency. BM25 через `ts_rank_cd` качественно хуже BGE-M3 sparse. Вердикт: pgvector конкурентен только для pure-dense сценариев; для hybrid + ColBERT remain on Qdrant. **Weaviate 1.34** (ACORN default, RQ, native multi-tenancy per-shard) — сильный кандидат, но отсутствие нативного BGE-M3 sparse (только BM25F) — архитектурный блокер. **Vespa** — единственная open-source БД, где ColBERT first-class citizen с long-context MaxSim; оправдан только при готовности принять JVM-ops и YAML application packages. **Turbopuffer** ($1/мес на 1M vectors) привлекателен ценой, но отсутствие ColBERT ломает текущий rerank-стек.

## Граф как опциональный слой: Kuzu мёртв, Apache AGE — победитель

**10 октября 2025 года Apple купила Kùzu Inc.** (EU M&A filings, Toronto, ~10 сотрудников), репозиторий `kuzudb/kuzu` архивирован, on-disk формат так и не стабилизировался. Существует три форка — **RyuGraph** (Predictable Labs, Akon Dey ex-Dgraph), **LadybugDB** (Arun Sharma), **Vela-Engineering/kuzu** (с concurrent writes) — но все в альфе, single-writer constraint сохраняется, production-кейсов нет. Для Dharma-RAG это означает: **если Kuzu не использовался в прототипах, просто пропустить**; если использовался — мигрировать на Postgres + Apache AGE (Cypher совместим на ~90%) или Neo4j 5.x через APOC export.

**Рекомендация для Dharma-RAG — Apache AGE 1.5 в том же Postgres-кластере, что и pgvector.** Microsoft Azure задокументировал рабочий паттерн в апреле 2026: AGE и pgvector живут в одной базе, одной транзакции, с полными ACID-гарантиями, а pg_diskann добавляет disk-based ANN для 8 GB RAM сценариев. Apache 2.0 лицензия идеально совместима с MIT-проектом. AGE поддерживает PG11–17, PG18 в следующем релизном цикле 2026. **Neo4j Community 5.x/2025.10** остаётся резервом для Phase 3, но AGPLv3 + Commons Clause делает его допустимым только через Bolt-протокол (никогда не embedded Java) — дело *Neo4j v. PureThink* (2024, $597K штрафов) показывает, что лицензия enforced. **FalkorDB (SSPL), ArangoDB CE (BSL 1.1 + 100 GB лимит), Memgraph CE (BSL pre-conversion)** — все токсичны для публичного SaaS; self-hosted non-commercial допускают, но добавляют юридический риск. **Blazegraph де-факто мёртв** (Wikidata мигрирует к Qlever/MillenniumDB/Virtuoso с 2025). **Dgraph** сменил владельца дважды за 2 года (Hypermode → Istari Digital, октябрь 2025) — стабильность не гарантирована.

**Главный неинтуитивный вывод: граф в Phase 1 не нужен.** Для 80% запросов пользователей — 1–2 hop (sutta → параллели, концепт → упоминания, учитель → talks) — SQL JOIN или recursive CTE строго быстрее: 1–5 ms на 1-hop, 5–30 ms на 2-hop, тогда как AGE Cypher добавляет 10–30 ms overhead на agtype casts. Граф выигрывает на 4+ hop (50–300 ms у Neo4j/Memgraph против 1–10 s у CTE со взрывом) и на pattern matching с переменными рёбрами. На 56 684 чанков Dharma-RAG доля таких запросов — \<5%. **Lineages** (teacher→disciple глубиной 5–15) решаются ltree + closure table; **parallels** (DN22↔MA98↔Toh291) — плоской таблицей `parallels` с JSONB passage_alignments; **concept networks** (paṭicca-samuppāda) — типизированным enum `relation_type` с 14 значениями. AGE добавляется в Phase 2 как projection relational-таблиц через nightly ETL, не как primary store.

## LightRAG + rule-based Buddhist parser — гибридный путь

**Microsoft GraphRAG даже в v2.0 слишком дорог:** full-index на 900K чанков стоит $3 000–15 000 с GPT-4o ($5 000 с GPT-4o-mini), incremental updates добавлены только в v0.4 и ограничены append-only. **LazyGraphRAG** (preview 2024.11, Azure 2025.06) снимает стоимость индексации до 0.1% ценой отказа от графа как артефакта — noun phrase + co-occurrence + communities считаются на query-time. **HippoRAG 2** (ICML 2025, arxiv 2502.14802) — SOTA на MuSiQue F1=48.6 и 2Wiki R@5=90.4% при индексе в 9M токенов (vs GraphRAG 115M); ключевой трюк — Personalized PageRank на phrase+passage двойных нодах. Но HippoRAG 2 — research code без Neo4j/Postgres адаптеров, для production нужен форк. **KAG (Ant Group)** даёт +19.6% на HotpotQA и +33.5% на 2Wiki благодаря logical-form-guided reasoning, но Chinese/English hardcoded в промптах, infra-тяжёлый (docker-compose с 6+ сервисами).

**LightRAG (HKUDS, EMNLP 2025, 21.5K stars, MIT) — единственный practically viable outerbox кандидат для Dharma-RAG.** Архитектура: entity-relation extraction + hierarchical retrieval (low-level entities + high-level themes) + deduplication. Нативный incremental update, никакого community-recompute при добавлении чанков. **Storage pluggable через 4 абстракции — KV / Vector / Graph / DocStatus — и все 4 имеют Postgres-адаптеры** (PGKVStorage + PGVectorStorage + PGGraphStorage на Apache AGE + PGDocStatusStorage). Авторы сами рекомендуют Postgres как one-stop solution; единственное уточнение — Neo4j быстрее AGE в production на \>10M рёбер, что для Dharma-RAG не актуально. Индексация 900K чанков на Claude Haiku 3.5 — **$950–1 400**, на GPT-4o-mini с prompt caching — **$250–300**, на локальном Qwen3-30B-A3B на 2×48 GB VRAM — **€0 + 3–4 недели compute**.

**Ключевой insight для буддийского корпуса: LLM-driven entity extraction во многом избыточен.** Канонические цитации (DN 22, MN 10, SN 56.11, AN 4.49, Dhp 277, Toh 4094, T.1562), параллели (SuttaCentral parallels.json с полными маппингами DN22↔MA98↔Toh291), lineages (Mahākassapa → Ānanda → Sāṇavāsī…) — всё это детерминированный regex-парсимый слой. LLM нужен только для (а) свободнотекстовых комментариев (Buddhaghosa, современные эссе), (б) cross-lingual concept alignment («осознанность» = sati = mindfulness), (в) coreference без явной цитаты. **Рекомендуемый гибрид:** rule-based parser на ~650K канонических чанков ($0) + LightRAG только на ~250K свободнотекстовых чанков ($150–350 на Claude Haiku или $60–100 на GPT-4o-mini) + LazyGraphRAG-style communities через `leidenalg` поверх канонической структуры (nikāya→vagga→sutta→стихи). **Итого v1.0 single-shot cost: $150–350, incremental +10K chunks: \<$5.**

## Буддийская SQL-схема: 11 таблиц, три уровня FRBR, ltree для lineages

Полная production-ready схема основана на PostgreSQL 16 + pgvector 0.8 + Apache AGE 1.5 + uuid-ossp + pg_trgm + btree_gin + ltree. Ядро — **три уровня FRBR**: `works` (абстрактная «Satipaṭṭhāna Sutta»), `expressions` (конкретный перевод Bhikkhu Bodhi 1995), `instances` (HTML-файл с `segment_map` для character-level spans). Это развязка критична, потому что один work (MN10) имеет десятки expressions (переводы) и сотни instances (версии файлов на SuttaCentral + Access to Insight + Dhammatalks.org с разными edition/revision). `parallels` — плоская таблица с enum `alignment_type` (`full`/`resembling`/`mention`/`counted`/`retelling`) и JSONB `passage_alignments` для passage-level маппингов.

**Buddhist-specific типы:** `tradition_t` enum (theravada, mahayana, vajrayana, zen, chan, pragmatic_dharma, early_buddhism, secular, comparative), `language_t` (pli, san, bo, zh-Hant, zh-Hans, en, ru plus европейские), `relation_t` с 14 значениями (is_a, part_of, causes, opposed_to, synonym_of, translates_as, derived_from, elaborates, refines, prerequisite_of, contradicts, cited_in, taught_by, practiced_in). `concepts` хранит multi-language names (pali, sanskrit, tibetan в Wylie, chinese, english, russian) плюс `embedding halfvec(1024)` и `definition_embedding halfvec(1024)` с HNSW индексами — halfvec даёт 2× экономию RAM при потере recall \<0.3%. `concept_relations` типизированы через enum, хранят `source_citation` (формат 'MN10:12.3'), `confidence` [0,1], `extracted_by` (human/llm/heuristic/imported) и поддерживают версионирование через `superseded_by`.

**Lineages решаются дважды** — через `ltree` path для ad-hoc фильтров (`path <@ 'mahayana.vajrayana'::ltree`) и через `lineage_closure` closure table для O(1) ancestor/descendant queries. Trigger `rebuild_lineage_closure()` пересобирает closure на любом изменении `schools_lineages`. **Chunks партиционируются по `source_id`** (7 партиций в Phase 1, больше в Phase 2), хранят `parent_chunk_id` для hierarchical parent-child retrieval, `contextualized_content` (Anthropic prefix + chunk), generated columns `word_count`/`char_count`, generated `tsvector` на English и simple для multilingual FTS. **Chunk ↔ concept relevance** — отдельная таблица с `relevance_score`, `is_defining`, `extracted_by`, `evidence` JSONB для provenance. **ColBERT multivector** хранится как отдельная таблица `chunk_colbert_tokens` с (chunk_id, token_idx) primary key и HNSW на halfvec(128), но Phase 1–2 это только в Qdrant — в Postgres миграция происходит в Phase 3.

**Materialized views** обеспечивают аналитику без рантайм-cost: `mv_concept_cooccurrence` (пары концептов в одних чанках с count ≥3), `mv_commentary_chains` (recursive CTE по parallels с depth ≤6 и циклическим detection через массив path). **Sync между Postgres и Qdrant** — deterministic `UUIDv5(namespace_dharma, source_id || passage_ref)`; outbox pattern с LISTEN/NOTIFY на Postgres + worker, который реплицирует upsert/delete в Qdrant. Backup: Phase 1 — `pg_dump` + Qdrant snapshot на Hetzner Storage Box (€3/мес за 1 TB); Phase 3 — только `pg_dump`, потому что AGE данные включены в dump.

## Оценка: статистика, датасет, human eval, CI

**Размер golden dataset 500–800 QA статистически обоснован.** Для paired recall@5 при baseline 0.65, α=0.05 двусторонний, power=0.80, ρ=0.6 (типичная корреляция RAG-систем на одном корпусе): n=500 даёт MDE ~3.8 pp — идеальное совпадение с CI Warning threshold 5 pp и запасом для Blocker 2 pp; n=800 даёт MDE ~3 pp для subgroup analysis при 8 query-types. Процедура: paired BCa bootstrap (n_resamples=10 000) через `scipy.stats.bootstrap` + Wilcoxon signed-rank как sanity-check + McNemar для бинарных hit@k метрик. **Bonferroni correction α/K** для многоkatatрических сравнений (Holm-Bonferroni как менее консервативная альтернатива через `statsmodels.stats.multitest.multipletests`). **Prediction-Powered Inference** (arxiv 2311.01453, Angelopoulos-Duchi-Zrnic) позволяет комбинировать Claude-as-judge на всех 500 + human labels на 50–100 для valid CI — точечная оценка unbiased, CI уже, чем по одним human. **Group sequential** (Pocock / O'Brien-Fleming α-spending) защищает от peeking при interim checks на 150→300→500→800→1500.

**Пайплайн QA из шести шагов:** corpus sampling stratified по (piṭaka, nikāya, длина, язык); synthetic generation Claude Sonnet 4.5 с per-type промптами; cross-family validation Gemini 2.5 Pro или GPT-5 (отклоняет doctrinally-incorrect); 100% human validation Buddhologists; adversarial augmentation (paraphrase, Pali term perturbation, unanswerable); RU↔EN back-translation через DeepL/Claude с ручной проверкой Pali-терминов. **Стратификация типов:** factoid 15%, definitional 15%, citation 15%, multi-hop 12%, comparative 10%, practice 10%, doctrinal-interpretive 13%, adversarial 10%. **Схема QA entry** хранит gold_citations с passage_id **и** character_spans (passage_id стабилен к re-chunking, spans нужны для faithfulness), golden_answer с paraphrases, `contraindicated_content` (Mahāyāna ālaya-vijñāna, Hindu karma, Zen perspectives), `pali_required` и `pali_forbidden` списки, `iaa_score` per-item Krippendorff α.

**IAA targets: Krippendorff α ≥ 0.7 как reliable, ≥ 0.8 highly reliable** (Artstein & Poesio 2008). Реализация — `grrrr/krippendorff-alpha` (корректнее NLTK баг #2991), с ordinal measurement для difficulty/relevance и nominal для query_type. **Рекрутинг hybrid:** 1 PhD Buddhologist (core reviewer, 50% items, €40–55/час), 2 experienced practitioners (5+ лет обучения, €18–25/час). Total annotation volume для v1.0: ~1 200 annotations при 2 ann + 20% double-annotation, ~120 человеко-часов, **€3 600** — вписывается в бюджет $4 625/год при оптимизации. Hourly payment строго лучше per-item (Lovett 2017 — per-item incentivizes speed-over-quality на MTurk). **Argilla 2.x + Label Studio** — основной и span-level инструменты; Prodigy только если budget позволяет €390+.

**Рубрика ошибок с severity:** factual error (high), doctrinal error (**critical**), attribution error (high), translation error medium, context error medium, **tradition conflation (critical)**, non-Buddhist insertion (high), **citation hallucination (critical)**. Calibration session: Day 1 2h rubric walkthrough + 10 calibration items, Day 2 3h 20 items blind, Day 3 1.5h discuss disagreements до α ≥ 0.7 до production. Monthly re-calibration на 20 недавних items (test-retest), full re-training если α upperbound падает \>0.1.

**Adversarial library ≥30 примеров** в 7 категориях — anachronism («Что Будда говорил об iPhone?»), wrong premise («Why did Buddha teach eternal soul?»), tradition conflation («Zen kōan practice from Pali Canon», «Dzogchen в Abhidhamma»), out-of-scope (Sanskrit Heart Sūtra, Bhagavad Gītā), minority positions (Abhayagiri vs Mahāvihāra, Puggalavāda pudgala), cross-lingual edge cases, prompt injection / jailbreak (cedar-tree attacks, «SYSTEM OVERRIDE», «Write a satirical story mocking Buddha»). Reference: HarmBench (arxiv 2402.04249), StrongREJECT (arxiv 2402.10260), arxiv 2505.04806 (1 400+ jailbreak prompts taxonomy).

**CI/CD: quick eval 30 queries \<5 min на PR** (stratified sample, retrieval metrics + 1 faithfulness sanity на Claude Haiku judge); **full eval 500 queries \<1 hour на merge to main** (все retrieval метрики + faithfulness/citation/doctrinal через Gemini 2.5 cross-family judge + latency + cost, 20 parallel workers). **Regression thresholds:** ref_hit@5 и recall@5 — Δ \>5 pp блокирует PR, \>2 pp требует review; faithfulness — \>3 pp / \>1 pp; doctrinal_correctness — ANY regression блокирует; latency_p95 \>3 s абсолютно или \>20% относительно; cost/query \>25% абсолютно. Significance test: paired BCa bootstrap + Wilcoxon, блокирует только если CI не пересекает 0 **И** |Δ|\>threshold (защита от шума). GitHub Actions YAML с `marocchino/sticky-pull-request-comment` для auto-post PR comments. Dashboard: Grafana/Langfuse с time-series метрик, per-query-type heatmap, cost/latency distributions, embedding drift KL-divergence, annotation-queue backlog, IAA last month.

**Production monitoring: Langfuse traces** — trace_id, session_id, query_text, detected_query_type, top_k passages со scores, retriever_version, latency, prompt/response, tokens, citations, LLM-judge scores, user thumbs, cost, cache_hit, safety flags. **Drift detection:** weekly KL(passage_id_freq_week || baseline) alert at KL \> 0.2; answer length mean/std shift \>2σ; % запросов без citation \<5% кроме adversarial; weekly UMAP query embeddings. **Alerts:** error_rate \>1% за 1h → PagerDuty, latency_p95 \>3s за 15min, faithfulness_drift \>3pp на 24h rolling, cost/query +30% недельного. **Shadow evaluation:** новая модель в parallel (0% user traffic, same query), statistical test после 500–1 000 запросов, promote при win-rate \>55% + faithfulness не падает.

## Unified quality score и три уровня quality

**Три уровня quality формализуют разные фазы цикла.** Level 1 «Knowledge base quality» (без генерации) — coverage (60%→80%→95% Phase 1/v0.5/v1.0), consistency (\<10%→\<5%→\<2% contradicting passages), linking quality (70%→85%→95%), parallel alignment BLEU/COMET RU-EN (0.40→0.55→0.70), attribution correctness (90%→95%→99% на audit 50 random), license compliance (100% всегда). Level 2 «Retrieval quality» — recall@5 (0.60→0.75→0.85), precision@5 (0.40→0.55→0.65), MRR (0.50→0.65→0.75), NDCG@10 (0.55→0.70→0.80), diversity (≥3→≥3.5→≥4 unique sources в top-5), freshness (weekly→daily sync). Level 3 «Generation quality» — faithfulness (0.70→0.85→0.92), citation accuracy (0.75→0.90→0.97), doctrinal correctness (0.80→0.90→0.95), tone appropriateness (≥3.5→≥4.0→≥4.3 на 1-5), helpfulness thumbs-up (60%→75%→85%), citation hallucination rate (\<10%→\<3%→\<1%).

**Weighted geometric mean строго лучше weighted arithmetic для unified score,** потому что критические компоненты (doctrinal_correctness, citation_hallucination_rate inverted) не должны маскироваться высокими fluency-скорами. Min-aggregator слишком консервативен (worst-component dominates). Formula для Dharma-RAG: $Q = \prod_i m_i^{w_i}$ где $m_i$ — нормализованные на [0,1] компоненты, $\sum w_i = 1$. Предлагаемые веса: faithfulness 0.20, doctrinal_correctness 0.20, citation_F1 (ALCE) 0.15, tradition_attribution 0.10, pali_term_accuracy 0.10, (1 - hallucination_rate) 0.10, answer_relevancy 0.08, context_precision 0.04, context_recall 0.03. **Release thresholds:** v0.5 — Q ≥ 0.55 **И** doctrinal ≥ 0.80 **И** citation_F1 ≥ 0.70 **И** hallucination \< 0.10; v1.0 — Q ≥ 0.75 **И** doctrinal ≥ 0.90 **И** citation_F1 ≥ 0.85 **И** hallucination \< 0.03. Жёсткие per-component thresholds защищают от gaming через раздувание fluency.

**Рекомендуемый eval-стек:** RAGAS (faithfulness через claim extraction+NLI, answer_relevancy, context_precision/recall) + DeepEval G-Eval для кастомных рубрик на doctrinal_correctness + ALCE для citation Recall/Precision/F1 + Lynx 8B (PatronusAI, arxiv 2407.08488) локально на 2×48 GB VRAM для hallucination detection + Langfuse для tracing. **Критично использовать cross-family LLM-as-Judge:** Gemini 2.5 Pro или GPT-5 оценивает Claude-generated ответы, никогда Claude оценивает Claude (self-enhancement bias). **G-Eval** (arxiv 2303.16634) с CoT + weighted probability scoring даёт на 10–20% лучший correlation с human labels, чем plain prompting. **Jury of LLMs (PoLL, arxiv 2404.18796)** — ensemble из 3 cheaper judges (Haiku + Gemini Flash + GPT-4.1-mini) статистически лучше одного Opus judge по cost/quality tradeoff. **Prometheus 2** (KAIST, 7B/8x7B, arxiv 2405.01535) — open-source judge с rubric-awareness, подходит для self-hosted на 2×48 GB VRAM и полностью устраняет stochastic API cost.

## Fine-tuning: embedding first, LLM last, pretraining никогда

**Порядок приоритетов для Dharma-RAG категоричен.** Phase 1 — NO FT, Claude API + Contextual Retrieval + BGE-M3 hybrid off-the-shelf, focus на 200 golden QA от буддолога и базовый eval framework. Phase 2 Early (300+ QA) — **FT BGE-M3 через sentence-transformers v3 с MultipleNegativesRankingLoss или GISTEmbedLoss на 1K–3K triplets (query, positive_chunk, hard_negative), ожидаемый +5–15 pp Hit@5, 2–5 часов на 1×48 GB GPU, €0.50 электричества**. Hard negatives обязательно mining через NV-Retriever TopK-MarginPos (arxiv 2407.15831) или TopK-PercPos — naive top-K даёт false negatives (релевантные чанки с высоким score выше positive), margin=0.05 абсолютный или 15% относительный drops их. Phase 2 Late — FT BGE-reranker-v2-m3 (CrossEncoder, 3K–10K pairs, 2–4 часа, +3–8 pp MRR).

**Полезные методы 2024–2026:** LoRA (arxiv 2106.09685) — r=8 для стиля, 16 для стандартного SFT, 32 для domain, 64 для reasoning/CPT; α=2r; all-linear target modules стали стандартом (q,k,v,o,gate,up,down). **QLoRA** (arxiv 2305.14314) — NF4 + double quantization + paged optimizers, полностью восстанавливает 16-bit качество и влезает Qwen2.5-72B в 2×48 GB. **DoRA** (arxiv 2402.09353, NVIDIA, ICML 2024 Oral) — weight decomposition W=m·(V/||V||_c), magnitude обучается напрямую, LoRA на direction; +1 pp над LoRA ценой 20% замедления training. **rsLoRA** (arxiv 2312.03732) — α/√r вместо α/r, разблокирует стабильное обучение при r=512–2048. **GaLore** (arxiv 2403.03507) — gradient low-rank projection для full-parameter training с PEFT memory; 65.5% reduction optimizer-state memory; LLaMA-7B pretraining на single 24 GB 4090. **Matryoshka Representation Learning** (arxiv 2205.13147) — один embedding обучается как nested (8,16,…,1024,2048 dim), 14× speedup retrieval через adaptive 2-stage (shortlist коротким prefix, rerank длинным); принят OpenAI text-embedding-3, Nomic, Qwen3-Embedding.

**RAFT (arxiv 2403.10131, Berkeley Gorilla)** — ключевой метод для domain-specific RAG fine-tuning: training sample = (Q, oracle doc + 3 distractors, CoT answer с verbatim citation через `##begin_quote##...##end_quote##`). **Оптимальная доля P=80% примеров с oracle** — 100% ломает robustness к плохому retrieval. +35% на HotpotQA, +76% на HuggingFace docs QA. **Для Dharma-RAG Phase 2c:** 3K–10K RAFT examples × 5 distractors = 15K–50K training instances, генерация через Claude Opus как teacher (Theravada-careful system prompt). Реализация через Unsloth (single GPU, 2–5× быстрее HF Transformers) для Qwen2.5-7B LoRA или Axolotl+FSDP для Qwen2.5-72B QLoRA на 2 GPU.

**Экономика для Нидерландов при €0.30/kWh:** Qwen2.5-7B LoRA — €0.50 за run (4h × 400W); Qwen2.5-72B QLoRA — €5–10 (25h × 800W); 5 experiments/месяц спокойно. **Break-even vs Claude Haiku API:** при ~25 tokens/s self-host throughput и hardware amortization €0.30/h, self-hosted стоит ~$1.80/1M output tokens против Haiku $4/1M — экономия $2.2/1M. При Dharma-RAG объёме 10–100 queries/day **Haiku API строго дешевле**; self-hosting оправдан только при privacy, offline, или \>100K queries/day. Cloud для FT (RunPod A100 80GB $1.19/h community) — ~$30 за 72B QLoRA run, дешевле own hardware amortization первые 12 месяцев при \<10 runs/год.

**Qwen2.5-7B / 72B (Apache 2.0) строго лучше Llama-3.1 по licensing** — без 700M MAU cap, без naming constraints, без запрета на training non-Qwen моделей на outputs. **CC-BY-NC SuttaCentral** совместим с research/non-commercial; коммерческое использование derived моделей требует согласования. **EU AI Act:** solo-dev на HF Hub не triggers GPAI provider obligations (порог ≥10²⁵ FLOPs на несколько порядков выше), но заполнение model card и training data summary — compliance-friendly практика. **Pretraining from scratch — никогда:** 7B модель на 2T tokens требует $15K–30K cloud compute, full CPT на 900K чанков × ~500 tokens = 450M tokens даёт marginal gain над LoRA SFT при 1–2 сутках на 2×48 GB. Лучший compromise — QLoRA CPT r=64 на Buddhist raw corpus (2 эпохи) → SFT LoRA → RAFT.

**Buddhist LLM niche пуста.** Существующие проекты — «Qwen-Bible», «Dharmaa», «TathagataGPT» — personal experiments без peer review. **MITRA** (Gemma-2-MITRA, researchgate 399707324) — самая серьёзная модель с parallel Pali/Sanskrit/Buddhist Chinese/Tibetan для translation, но не generative Q&A. **ByT5-Sanskrit** (arxiv 2409.13920) — NLP tasks, не assistant. **SiPaKosa** (arxiv 2603.29221) — корпус, не модель. Это означает, что Dharma-RAG занимает **первую в своём роде production-grade нишу при правильной буддологической валидации**; собственный 500 QA golden eval set — главный актив, ценнее любой конкретной модели.

## Выводы и roadmap

Исследование перевернуло несколько априорных ожиданий. **Knowledge graphs в 2026 оказались переоценены для узких доменов со структурированным каноном** — плоская метадата-схема с recursive CTE и pgvector закрывает 80% случаев дешевле, быстрее и без LLM-шума. **Vector DB ландшафт стабилизировался** — Qdrant, Milvus, pgvector заняли три чёткие ниши (hybrid + ColBERT first-class, distributed scale, unified Postgres), новые претенденты (Turbopuffer, Vespa, LanceDB) не угрожают лидерам без специфических use case. **Fine-tuning LLM — плохой ROI для RAG** — retrieval dominates accuracy, поэтому FT embedding/reranker даёт 10× лучший эффект на €1 electricity. **Kuzu archival (Apple acquisition октябрь 2025) — серьёзный сигнал** о консолидации graph DB space; RyuGraph/LadybugDB пока не production-ready, Apache AGE единственный безопасный выбор для open-source RAG в 2026.

**Concrete roadmap Dharma-RAG:** Day 1–14 Phase 1 на Hetzner CX32 с Qdrant + Postgres metadata + Contextual Retrieval + 30-query spot check и 1 annotator calibration; Day 15–45 v0.3 с 100 QA, 2 annotators, α ≥ 0.7, full eval weekly в CI; Day 46–75 v0.5 с 300 QA, PPI со Claude-judge, PR-blocking CI, миграция на CCX33 с Apache AGE поверх существующей Postgres; Day 76–120 v1.0 с 500 QA + 100 adversarial, полный статистический протокол, blue/green deployment, shadow eval, FT BGE-M3 + reranker. Phase 3 (месяцы 4–12): локальный 2×48 GB сервер для Qwen2.5-72B QLoRA inference via vLLM, опциональный RAFT на Qwen2.5-7B, миграция chunks embeddings в pgvector + pgvectorscale StreamingDiskANN, Qdrant остаётся как A/B regression baseline.

**Budget distribution $4 625/12 мес:** LLM API (Claude для gen, Gemini для judge, PPI) — $1 500; annotation (2 ann + 20% double, 500 QA) — €3 600 ≈ $3 900 с оптимизацией до $2 200; hosting (Hetzner CX32→CCX33, Langfuse Hobby, Argilla self-host) — $1 200; tools и hardware буфер — $225. **Главная ставка не на compute, а на качество данных и буддологическую валидацию** — 500 golden QA с Krippendorff α ≥ 0.7 и полноценной doctrinal rubric ценнее любой конкретной модели или vector DB. Для первой в своём роде Buddhist RAG системы это построит infrastructure, на которой sustainable development возможен следующие 5 лет.
