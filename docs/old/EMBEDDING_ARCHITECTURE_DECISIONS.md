# Архитектура эмбеддингов: от выбора модели до продакшн-деплоя

Построение гибкой системы векторного поиска требует одновременного решения пяти взаимосвязанных задач: выбор модели эмбеддингов, квантование, хранилище, граф-возможности и архитектура миграций. Ключевой вывод исследования — **Qwen3-Embedding-0.6B в квантизации Q8_0 при 639 МБ VRAM уже превосходит большинство 7B-моделей** на бенчмарке MTEB, а PostgreSQL с pgvector + Apache AGE может заменить связку «Qdrant + Neo4j» для проектов до 10M векторов, обеспечивая транзакционную консистентность и единый бэкап. При этом квантование весов модели до INT8/Q8_0 практически бесплатно (<1% потери качества), тогда как квантование самих векторов в базе данных — более рискованный, но управляемый компромисс.

---

## Qwen3-Embedding: три размера, один архитектурный выбор

Серия Qwen3-Embedding, выпущенная 5 июня 2025 года, включает три модели с поддержкой Matryoshka Representation Learning (MRL), позволяющей гибко выбирать размерность эмбеддинга без переобучения.

| Характеристика | 0.6B | 4B | 8B |
|---|---|---|---|
| Размерность по умолчанию | 1024 | 2560 | 4096 |
| Диапазон MRL | 32–1024 | 32–2560 | 32–4096 |
| Макс. длина контекста | 32K | 32K | 32K |
| MTEB Multilingual (mean) | 64.33 | 69.45 | **70.58** (#1) |
| MTEB Retrieval | 64.64 | 69.60 | 70.88 |
| VRAM (BF16) | ~1.2 ГБ | ~8.0 ГБ | ~15.1 ГБ |
| VRAM (Q8_0) | ~639 МБ | ~4.3 ГБ | ~8.1 ГБ |
| VRAM (Q4_K_M) | ~395 МБ | ~2.5 ГБ | ~4.6 ГБ |
| GGUF-файл (Q4_K_M) | 395 МБ | 2.48 ГБ | 4.64 ГБ |

**Модель 0.6B — главный сюрприз серии.** С результатом **64.33 на MTEB Multilingual** она обходит NV-Embed-v2 (7.8B, 56.29), gte-Qwen2-7B (7.6B, 62.51) и text-embedding-3-large от OpenAI (58.93). В сценариях с ограничением латентности (<10 мс на запрос) или на edge-устройствах это единственный разумный выбор: Q4_K_M весит всего 395 МБ и работает на CPU. При этом разрыв между 0.6B и 8B составляет ~6 пунктов MTEB — значительный для задач retrieval, но некритичный для многих RAG-пайплайнов, где качество ответа определяется прежде всего реранкером и LLM.

**Когда 0.6B выигрывает у 8B на практике.** При обработке больших корпусов документов модель 0.6B обеспечивает ~13x больший throughput на GPU-секунду. Для batch-индексации миллионов документов это означает разницу между часами и днями работы. На CPU модель 0.6B в Q4_K_M — единственная из серии, способная работать с приемлемой латентностью (~50–100 мс/запрос). Модель 8B на CPU нецелесообразна.

GGUF-форматы доступны как от Qwen (официальные: Q4_K_M, Q5_0, Q5_K_M, Q6_K, Q8_0), так и от сообщества (Mungert: полный диапазон от IQ2_XXS до BF16). Ollama поддерживает все три модели из коробки (`ollama pull qwen3-embedding`). Инференс доступен через Sentence Transformers (≥2.7.0), vLLM (≥0.8.5, `task="embed"`), TEI и llama.cpp.

## Qdrant: именованные векторы решают проблему мультимодельности

Qdrant полностью поддерживает коллекции с разными размерностями векторов на одном сервере. Но ещё важнее механизм **named vectors** — возможность хранить несколько эмбеддингов разных моделей для каждого документа в одной коллекции.

```python
client.create_collection(
    collection_name="knowledge_base",
    vectors_config={
        "qwen_0.6b": models.VectorParams(size=1024, distance=models.Distance.COSINE),
        "qwen_8b":   models.VectorParams(size=4096, distance=models.Distance.COSINE),
    }
)

# Поиск по конкретной модели
results = client.search(
    collection_name="knowledge_base",
    query_vector=("qwen_0.6b", query_embedding),
    limit=10
)
```

Каждый именованный вектор имеет **собственную конфигурацию HNSW и квантования**, что позволяет, например, хранить быстрый 1024-мерный вектор со scalar quantization для первичного поиска и полный 4096-мерный для финального ранжирования. Qdrant поддерживает двухпроходной поиск через `prefetch`: сначала быстрый кандидатный поиск по компактному вектору, затем rescoring по полному.

**Потребление памяти.** Формула Qdrant: `memory = vectors × dimension × 4 bytes × 1.5`. Для 1M векторов: 768d ≈ 4.6 ГБ, 1024d ≈ 6.1 ГБ, 4096d ≈ 24.6 ГБ. WAL каждого шарда потребляет по умолчанию 32 МБ, плюс базовые ~8 ГБ на системные нужды. **Рекомендация Qdrant — избегать создания множества мелких коллекций** (это явно названо антипаттерном); вместо этого используйте payload-based multitenancy или named vectors.

**Стратегия миграции моделей.** Qdrant v1.16 добавил conditional updates специально для blue-green миграций эмбеддингов. Рекомендованный подход:

1. Добавить новый named vector к существующей коллекции
2. Пакетно переэмбеддить документы (batch по 50–100)
3. Валидировать результаты поиска по новому вектору на «золотых запросах»
4. Переключить запросы через feature flag
5. Удалить старый вектор после недели стабильной работы

**Встроенная квантование в Qdrant** включает три метода: scalar (INT8, 4x сжатие, <1% потери recall), binary (1-bit, 32x сжатие, требует ≥1024d и oversampling), product (до 64x, но медленнее scalar). С версии 1.15 доступны промежуточные варианты: 1.5-bit (24x) и 2-bit (16x). Квантование конфигурируется per-vector, а переключение метода не требует перезагрузки данных.

## PostgreSQL как единое хранилище: pgvector + Apache AGE

Использование PostgreSQL с расширениями pgvector и Apache AGE создаёт архитектуру «всё в одной базе», где эмбеддинги, структурированные данные и графовые связи живут в одном транзакционном контексте.

**pgvector (v0.8.2)** поддерживает четыре типа векторов: `vector` (float32, до 2000 индексируемых измерений), `halfvec` (float16, до 4000), `bit` (бинарный, до 64000) и `sparsevec` (разреженный). Это даёт встроенную поддержку scalar и binary квантования через expression indexes:

```sql
-- Полнопрецизионное хранение, но HNSW-индекс на halfvec
CREATE INDEX ON documents
  USING hnsw ((embedding::halfvec(768)) halfvec_cosine_ops)
  WITH (m = 16, ef_construction = 256);
```

**Производительность pgvector в сравнении с Qdrant** зависит от масштаба. Бенчмарк TigerData на **50M векторов** (768d, AWS r6id.4xlarge) показал удивительный результат:

| Метрика (recall 99%) | PostgreSQL + pgvectorscale | Qdrant |
|---|---|---|
| p50 латентность | 31.07 мс | **30.75 мс** |
| p95 латентность | 60.42 мс | **36.73 мс** |
| Throughput (QPS) | **471.57** | 41.47 |

Qdrant выигрывает по tail-латентности (**39% лучше на p95**), а pgvectorscale — по throughput (**11.4x выше**). На малых масштабах (<5M векторов с HNSW) разница практически неразличима — обе системы выдают ~5 мс на запрос. Для большинства RAG-приложений с корпусом до нескольких миллионов документов pgvector — достаточный выбор, экономящий операционные затраты.

**Apache AGE** — расширение, превращающее PostgreSQL в графовую базу данных с поддержкой языка Cypher. Каждый граф хранится как PostgreSQL-схема, где метки вершин и рёбер становятся таблицами. Это позволяет **объединять Cypher-запросы с SQL через JOIN**:

```sql
-- Семантический поиск + граф цитирований в одном запросе
WITH similar_docs AS (
  SELECT id, 1 - (embedding <=> $query_vector) as similarity
  FROM documents
  ORDER BY embedding <=> $query_vector
  LIMIT 20
)
SELECT sd.*, cg.target, cg.depth
FROM similar_docs sd
LEFT JOIN cypher('citation_graph', $$
  MATCH (d:Document)-[:CITES*1..3]->(cited:Document)
  RETURN d.doc_id, cited.doc_id, length(relationships)
$$) as cg(source agtype, target agtype, depth agtype)
ON sd.id = cg.source::int;
```

Microsoft использует именно эту архитектуру (AGE + pgvector + реляционные данные) в своём **GraphRAG Solution Accelerator** для Azure PostgreSQL. Для ограниченных запросов (2–3 уровня обхода) производительность AGE сопоставима с нативными графовыми БД. Глубокие неограниченные обходы (>5 уровней) на миллионах узлов — территория Neo4j и Memgraph.

**Альтернативы AGE.** Рекурсивные CTE справляются с простыми графовыми обходами на малых графах (<50K рёбер) за микросекунды, но код быстро становится нечитаемым. Расширение `ltree` отлично подходит для статических иерархий (таксономии, оргструктуры) с GiST-индексацией. `pg_graphql` — не графовая база, а генератор GraphQL API от Supabase. Стандарт SQL/PGQ (часть SQL:2023) обещает нативные графовые запросы, но для PostgreSQL пока доступен только как экспериментальные патчи.

**Главное преимущество единой базы — ACID-консистентность.** Вставка эмбеддинга, обновление метаданных и создание графового ребра происходят в одной транзакции. Нет eventual consistency между базами, нет слоя синхронизации, один бэкап, один мониторинг, одна строка подключения.

## Архитектура гибкости: как менять модели без переиндексации

Полностью избежать переэмбеддинга при смене модели невозможно — разные модели создают несовместимые векторные пространства. Но правильная архитектура минимизирует простой и риск.

**Абстракция модели эмбеддингов.** Минимально достаточный интерфейс включает два метода (`embed_documents`, `embed_query`) и два свойства (`dimension`, `model_name`). LangChain и LlamaIndex предоставляют готовые базовые классы. Критический совет из продакшн-опыта: **вынесите модель и размерность в переменные окружения** — `EMBED_MODEL` и `EMBED_DIMENSIONS`. Разница между двухдневной и двухнедельной миграцией часто определяется этими двумя строками конфигурации.

**Matryoshka Representation Learning (MRL)** — ключевая технология для гибкости размерности. Все три модели Qwen3-Embedding поддерживают MRL: любой префикс вектора (от 32 до максимальной размерности) является полноценным эмбеддингом. OpenAI text-embedding-3-large при усечении с 3072 до 256 измерений **превосходит** старую text-embedding-ada-002 (1536d). Это открывает двухэтапный поиск: быстрый coarse search по 128-мерным MRL-эмбеддингам → rescoring по полным эмбеддингам.

**Стратегия zero-downtime миграции** (проверенная в продакшене при переходе text-embedding-004 → text-embedding-3-large):

1. Добавить новый столбец/named vector (не заменять старый)
2. Пакетно переэмбеддить с чекпоинтами и хешированием контента (пропуск неизменённых документов)
3. Валидировать: сравнить результаты поиска по 100+ тестовым запросам (ожидайте ~82% пересечения — разные модели дают разные пространства)
4. Переключить через feature flag (`USE_V2_EMBEDDINGS=true`)
5. Удалить старые данные через неделю стабильной работы

**Опыт Notion** демонстрирует масштабную миграцию: они используют генерационное шардирование, где новые поколения индексов создаются с обновлённой моделью, миграция происходит поколение за поколением с валидацией, а **content hashing** (отдельные хеши для текста и метаданных) предотвращает избыточное переэмбеддирование.

**Кэширование эмбеддингов** организуется в три слоя: (1) кэш «текст→вектор» в Redis с ключом `hash(model_name + normalized_text)` — убирает 50–200 мс латентности; (2) кэш «запрос→document_ids» для повторяющихся запросов; (3) семантический кэш (Redis SemanticCache) для похожих запросов — до 15x быстрее и 90% снижения стоимости LLM-вызовов, но с риском ложных попаданий. **В ключ кэша обязательно включайте имя модели** — иначе получите кросс-модельное загрязнение.

## Квантование и качество поиска: где проходит граница

Квантование применяется на двух уровнях: к **весам модели** (GGUF, GPTQ, INT8) и к **выходным векторам** (scalar, binary, product в базе данных). Эти два уровня независимы и кумулятивны.

**Квантование весов модели** — почти бесплатная оптимизация. Q8_0 даёт +0.0009 к perplexity (0.01% деградация) при 50% экономии памяти. Q4_K_M увеличивает perplexity на ~0.7% — приемлемый компромисс для большинства задач. NVIDIA продемонстрировала, что INT8-квантование NV-Embed (7B) сохраняет **0% потери MTEB-качества**. Intel подтверждает: **<1.55% ошибки на MTEB Retrieval** для INT8-квантованных BGE/GTE моделей. Критическое наблюдение: **малые модели (<3B) более чувствительны к квантованию** — Qwen-3B теряет 0.14%, тогда как Minitron-4B теряет 0.84%.

**Квантование выходных векторов** — более рискованная, но более эффективная оптимизация:

| Метод | Сжатие | Потеря качества | Восстановление с rescoring |
|---|---|---|---|
| bfloat16/float16 | 2x | **~0%** | Не требуется |
| float8 | 4x | **<0.3% nDCG@10** | Не требуется |
| Scalar (INT8) | 4x | ~1–1.5% | До ~99% с rescore ×4 |
| Binary (1-bit) | 32x | 7–18% | До ~96–99% с rescore ×4 |
| Product (PQ) | 24–64x | 10–30% | До ~90% с rescoring |

Прорывной результат 2025 года: **float8-квантование векторов превосходит INT8 при том же сжатии 4x** — <0.3% потери nDCG@10 против ~1.5% для INT8 (Huerga-Pérez et al., HAIS 2025). Более того, float8 проще в реализации (не нужна калибровка). Это ставит под сомнение индустриальный стандарт дефолтного INT8.

**Бинарное квантование работает только для эмбеддингов ≥1024 измерений.** Qdrant демонстрирует **0.98 recall@100** для OpenAI ada-002 (1536d) и Cohere (4096d) с oversampling ×4. Но при 384 измерениях Recall@10 падает с 0.481 до 0.391 (без rescoring), а при 128d — катастрофически до 0.242. Модели на архитектуре ModernBERT показывают лучшую устойчивость к бинарному квантованию (98% качества против 87% у E5-small-v2).

**Комбинация MRL + квантование** — самая мощная стратегия сжатия. MRL с 1024 до 128 измерений + scalar INT8 = **32x сжатие при ~3% потере качества**. MRL до 128d + binary = 256x сжатие, но потери достигают ~10% — приемлемо для первичного этапа двухфазного поиска.

**Практическая рекомендация: три уровня оптимизации**, применяемые последовательно:

- **Шаг 1**: BFloat16 для хранения векторов — 2x сжатие, 0% потерь (делать всегда)
- **Шаг 2**: Q8_0 для весов модели — 50% экономии VRAM, <0.01% потерь (делать всегда)
- **Шаг 3**: Scalar INT8 в базе данных с rescore ×4 — дополнительные 4x при <1.5% потерь (делать при необходимости)

## Заключение: какую архитектуру выбрать

Исследование выявило несколько неочевидных выводов, меняющих подход к проектированию.

**Qwen3-Embedding-0.6B — новый baseline.** При 639 МБ в Q8_0 эта модель превосходит большинство 7B-моделей и достаточна для 80% RAG-задач. Модель 8B оправдана только для мультиязычных корпусов или задач, где каждый процент recall критичен. Начинайте с 0.6B и переходите на 8B только при доказанной необходимости.

**PostgreSQL может заменить два инструмента одним.** Для проектов до 5–10M векторов pgvector с HNSW выдаёт ~5 мс на запрос — идентично Qdrant. С pgvectorscale (StreamingDiskANN) масштабирование расширяется до 50–100M. Apache AGE добавляет графовые возможности с Cypher-синтаксисом. Единая транзакция для векторов, метаданных и графовых связей устраняет целый класс проблем синхронизации.

**Float8 > INT8 для квантования векторов** — это свежий и важный вывод. При одинаковом 4x-сжатии float8 даёт <0.3% потерь против ~1.5% для INT8, и не требует калибровки. Индустрии стоит пересмотреть дефолт. А бинарное квантование в Qdrant с oversampling — мощный инструмент для моделей с размерностью ≥1024, дающий 32x сжатие при ~2% потерь.

**Named vectors в Qdrant — архитектурный ключ к гибкости.** Хранение эмбеддингов нескольких моделей в одной коллекции с per-vector конфигурацией HNSW и квантования делает A/B-тестирование и миграцию моделей операционно тривиальными. В PostgreSQL аналогичный подход реализуется через дополнительные столбцы с expression indexes. Оба подхода работают — выбор определяется тем, готовы ли вы управлять дополнительным сервисом.
