# 07 — RRF hybrid fusion

## Что это

**RRF** = **Reciprocal Rank Fusion** = «слияние через обратные ранги».
Алгоритм 2009 года (Cormack et al.) для **объединения результатов из
нескольких поисковиков** в один ранжированный список.

Один поисковик дал свой top-N, второй — свой, третий — свой. Как
объединить, если у каждого **своя шкала**? RRF отвечает: **игнорируй
скоры, работай только с рангами**. Документ, который у dense на 1-м
месте, у sparse на 5-м, у BM25 не появился вообще — получает один
итоговый score. Документ, который у всех трёх в топ-3 — получает
гораздо больший score и поднимается выше.

## Формула в одну строку

```
RRF(d) = Σ по всем каналам c:  1 / (k + rank_c(d))
```

- `d` — документ
- `c` — канал (dense / sparse / BM25)
- `rank_c(d)` — позиция документа в топе канала `c` (1, 2, 3, ...). Если документа в топе нет — слагаемое = 0.
- `k` — константа из статьи. **k=60** — стандарт.

## Зачем у нас

После day-10 (Qdrant dense+sparse) и day-11 (BM25) у нас **три
независимых ранжированных списка**:

```
dense:   [SN56.27, SN56.29, SN56.28, MN141, AN3.61, ...]   (по cosine ~0.7)
sparse:  [SN56.13, SN56.28, AN3.61, SN56.27, ...]          (по dot product ~3.0)
BM25:    [SN56.29, SN56.23, MN141, SN56.15, ...]           (по ts_rank_cd ~0.5)
```

**Шкалы несопоставимы.** Cosine ∈ [-1, 1]. ts_rank_cd ∈ [0, 2+]. Sparse
∈ [0, 3+]. Если просто сложить — BM25 (с самой низкой шкалой) утонет.
Min-max нормализация требует знать распределение по корпусу. Z-score
требует средних. Всё это шумно.

**RRF не нормирует** — он работает на рангах. Просто. Робастно.

## Как работает (на нашем примере)

Запрос «four noble truths». Каналы вернули:

```
dense:   [SN56.27, SN56.29, SN56.28, MN141, AN3.61]
sparse:  [SN56.13, SN56.28, AN3.61,  SN56.27]
BM25:    [SN56.29, SN56.23, MN141,   SN56.15]
```

Считаем RRF (k=60):

| Документ | dense rank | sparse rank | BM25 rank | RRF score |
|---|---|---|---|---|
| **SN56.29** | 2 | — | 1 | 1/62 + 0 + 1/61 = **0.03252** |
| **SN56.27** | 1 | 4 | — | 1/61 + 1/64 + 0 = **0.03202** |
| **SN56.28** | 3 | 2 | — | 1/63 + 1/62 + 0 = **0.03200** |

**Итог:** SN56.29 (попавший в 2 канала на топовых позициях) обогнал
SN56.27 (тоже в 2 каналах, но хуже позиции) и SN56.28. Это и есть
**магия fusion**: «приз за широкое присутствие», не «за победу в одном
канале».

## Почему k=60

**k** — «выравниватель»:

| k | rank-1 score | rank-2 score | rank-30 score | Эффект |
|---|---|---|---|---|
| 1 | 0.500 | 0.333 | 0.032 | Топовая позиция доминирует |
| **60** | **0.0164** | **0.0161** | **0.0111** | **Все позиции близки, важно «быть в топе вообще»** |
| 1000 | 0.0010 | 0.0010 | 0.0010 | Все позиции почти равны (плохо) |

`k=60` — empirical sweet spot из статьи Cormack 2009. Все
индустриальные системы (Elasticsearch, Pinecone, Vespa) используют 60
как дефолт.

## Почему ровно так, а не score-averaging

| Подход | Проблема |
|---|---|
| `mean(score_dense, score_sparse, score_BM25)` | Шкалы разные. BM25 утонет. |
| `mean(min_max_normalize(...))` | Требует знать min/max по корпусу. Зависит от запроса. |
| `mean(z_score(...))` | Требует среднего и std. Тоже per-corpus. |
| **RRF** | **Работает только на рангах. Без статистики. Любой запрос.** |

## Параметры в нашем pipeline

| Параметр | Значение | Где |
|---|---|---|
| `k` | 60 | константа статьи |
| `per_channel_limit` | 30 | каждый канал даёт top-30 |
| `top_k` | 20 | RRF возвращает top-20 после слияния |
| Веса каналов | 1, 1, 1 (равные) | пересмотрим после day-14 eval |

30 на канал → объединение даёт ~50-90 уникальных кандидатов (зависит
от пересечения). Из них фильтруем по RRF в top-20.

## Результат на нашем корпусе

Smoke-тесты (день 12, 6 478 чанков, 10 канонических запросов):

- `mindfulness of breathing` → **MN 118** [3/1/1] — все три канала
  согласились
- `four noble truths` → **SN 56.29** [2/5/1]
- `Anāthapiṇḍika` → **SN 2.20** [1/1/2] — все три согласились
- `satipaṭṭhāna` → **AN 10.67** [10/12/·] — **никакой канал в одиночку
  его в топ не ставил, но fusion поднял** благодаря согласию dense+sparse
- `страдание` (RU) → **SN 35.180-182** [1/·/·] — только dense

Latency на 1080 Ti: **62-96 ms/query** end-to-end (encode 46-53,
channels 14-46 параллельно, fusion <1, enrich 1-11).
