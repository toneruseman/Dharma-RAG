# 21 — Confidence indicator для chat-ответа (app-day-24)

> **Статус:** реализовано в app-day-24. Под каждым ответом в чате
> появляется цветной бейдж — короткая метка вида `well-grounded`,
> `synthesized`, `limited grounding`, `interpretive — verify with
> teacher` или `no sources`. Это **confidence indicator** —
> визуальный показатель «насколько верить ответу», основанный на
> простой структурной арифметике (сколько цитат нашли и где они
> стоят), без нейросетей и без понимания смысла.

## Зачем

LLM — большая языковая модель (large language model), та самая
которая генерирует наш ответ — пишет одним и тем же уверенным
тоном независимо от того, опирается она на 5 найденных источников
или вообще ни на один. Текст звучит складно в обоих случаях.
Пользователь не отличит «диссертацию со ссылками на каждое
утверждение» от «мнения за чашкой кофе» — оба выглядят одинаково
авторитетно.

Для буддийских текстов это особенно опасно. Если модель додумывает
доктрину (интерпретирует учение от себя, а не цитирует канон),
читатель должен **сразу видеть**, что ответ нужно перепроверить с
учителем или живым источником. Без явного сигнала «осторожно» он
поверит сгенерированному тексту как канону.

Аналогия: у газеты на каждой статье стоит штамп — «надёжный
источник», «спорное», «мнение редакции». Читатель за секунду
понимает уровень доверия и не путает аналитику с колонкой
комментатора. Confidence indicator у нас — тот же штамп, только
автоматический.

Так делают и в индустрии. Glean, Perplexity, NotebookLM — у всех
под ответом виден сигнал доверия. Зелёный = можно опираться;
красный = проверь.

Главный вопрос — **как** считать этот сигнал. У нас нет размеченных
данных «вот тут ответ хороший, вот тут плохой» — а без них нельзя
обучить нейросеть-классификатор. Поэтому делаем простую структурную
эвристику и калибруем пороги потом, когда соберём пользовательский
feedback (👍/👎-виджет в app-day-26).

## Что такое confidence indicator — простыми словами

Это короткая цветная плашка под ответом. У неё три части:

- цветная точка слева (зелёная / жёлтая / оранжевая / красная)
- крупная заглавная подпись — название уровня (например
  `WELL-GROUNDED`)
- одна строка пояснения — почему именно этот уровень

Уровень мы называем словом **tier** — буквально «ступень» или
«ярус». Всего пять tier'ов, отсортированных от самого надёжного
к самому опасному. Бейдж не объясняет, *хороший* ли ответ по
смыслу — он отвечает только на вопрос «насколько хорошо ответ
подкреплён найденными источниками».

## Архитектура

Поток данных при отрисовке ответа:

```
chat/page.tsx (страница чата)
  ├─ получает ответ от backend: { answer, sources }
  └─ useMemo(() => computeConfidence(answer, sources))
       ↓ возвращает { tier, label, reason }
       ↓
       <ConfidenceBadge tier=... label=... reason=... />
           ├─ цветная точка 8px (dot)
           ├─ label заглавными (font-semibold)
           └─ reason — короткая фраза «почему этот tier»
```

Что здесь происходит. Страница чата получает от бэкенда два поля —
`answer` (текст) и `sources` (список найденных в Qdrant работ).
Эти два значения идут в **pure-функцию** `computeConfidence`. Pure
function (чистая функция) — это функция без побочных эффектов:
одинаковый вход всегда даёт одинаковый выход, она не пишет в файлы,
не дёргает сеть, ничего вне себя не меняет. Удобно: легко тестировать,
можно безопасно мемоизировать.

`useMemo` — это **React-хук кеширования**. Хук — специальная
функция React, которая подключается к жизненному циклу компонента.
`useMemo` запоминает результат вычисления и возвращает его на
следующих рендерах **без пересчёта**, пока зависимости не поменялись.
Аналогия: «закладка с готовым ответом — пока учебник не переписали,
не считаем заново». В нашем случае confidence пересчитывается только
когда меняется сам ответ, а не на каждый рендер компонента.

Результат — объект `{ tier, label, reason }` — отдаётся в
`<ConfidenceBadge>`, который рисует плашку.

## Логика и tier'ы

Функция `computeConfidence` смотрит на два структурных признака.

**Первый — количество уникальных citations.** Citation в нашем
случае — это вставка вида `[mn10]` или `[sn56.11]` в тексте ответа.
Парсер `parseAnswerCitations` (тот же, что используется в
`AnswerView` для подсветки) проходит regex'ом по тексту и собирает
все work_id из квадратных скобок. Дубликаты не считаются: если
модель сослалась на `[mn10]` пять раз, это всё равно один уникальный
источник.

**Второй — citation distribution (распределение цитат по тексту).**
Считаем позицию **последней** цитаты относительно длины ответа.
Если она находится после 60% длины — citations распределены по
тексту. Если все они в первой половине, говорим что ответ
**frontload'ed** (всё в начале, потом ничего).

Аналогия для frontload: школьное сочинение, в котором первый
абзац аккуратно списан с учебника со ссылками, а дальше — три
страницы «от балды». Введение выглядит академично, но содержательная
часть ничем не подкреплена. Слабый сигнал.

Из этих двух чисел получаем tier по таблице:

| Tier | Условия | Цвет (Tailwind) | Расшифровка |
|---|---|---|---|
| `well-grounded` | ≥3 уникальных citations + последняя ≥60% длины | emerald (зелёный) | надёжно, можно опираться |
| `synthesized` | ≥2 уникальных citations | amber (жёлтый) | синтез из нескольких — сверь с источниками |
| `limited` | ровно 1 уникальная citation | orange (оранжевый) | узкая база |
| `interpretive` | 0 citations в тексте, но sources были найдены | rose (розово-красный) | модель говорит вне найденных текстов |
| `no-sources` | retrieval вернул пустой список | destructive (красный) | модель отказалась отвечать |

Цвета — это **семейства Tailwind** (Tailwind — CSS-фреймворк, в
котором цвета сгруппированы в шкалы по 10 оттенков). `emerald`,
`amber`, `orange`, `rose` — это именованные палитры, у каждой
есть варианты от 50 (почти белый) до 950 (почти чёрный). Мы
берём средние оттенки для светлой темы и более яркие для тёмной
через модификатор `dark:text-emerald-300`. `destructive` — это
наш токен из shadcn/ui, который маппится на красный.

**Threshold** (пороговое значение для классификации) у нас три:
3 citations для well-grounded, 2 для synthesized, 60% длины для
distribution check. Почему именно эти цифры? Здравый смысл плюс
наблюдение за stub-режимом: 3 источника — это «достаточно широкая
база», 60% — «citations явно дотянулись до второй половины ответа,
а не остались во вступлении».

Это стартовая **calibration** (настройка порогов под реальные
данные). Аналогия: весы на почте сначала откалибровали в заводских
условиях, потом подкручивают под реальные посылки которые приходят.
Сейчас у нас «заводская» калибровка. После сбора 👍/👎-feedback'а
в app-day-26 мы посмотрим, как пользователи реально оценивают
ответы с 2 vs 3 citations, и подвинем пороги.

### Defensive cases (что делаем в крайних случаях)

- Пустой ответ или `sources=[]` → сразу `no-sources`. Тот же путь
  что в `<AnswerView>` со fallback-сообщением «модель отказалась
  отвечать».
- **Hallucinated citation** — модель придумала `[mn99]`, которой
  нет в найденных sources. Парсер такие отбрасывает, в подсчёт
  они не идут. Это правильно: ссылка на несуществующий источник —
  плохой сигнал, а не хороший. Не повышаем tier за галлюцинацию.

## Ключевые решения

### 1. Структурная эвристика, не ML-классификатор

**Heuristic** — это «правило большого пальца», грубое практическое
правило вместо точной формулы. **Structural heuristic** = считаем
структурные признаки текста (число цитат, их позиции), не пытаясь
понять смысл.

Альтернатива — ML-классификатор. **ML classifier** — нейросеть,
которую обучают на размеченных примерах присваивать классы новым
данным. Например, **FactScore** — научная метрика, оценивающая
«насколько утверждения текста подкреплены источниками»: разбивает
ответ на atomic claims, проверяет каждый по корпусу, выдаёт число
от 0 до 1. Считается через отдельную LLM-проверку, дорого и сложно.

Аналогия: считаем галочки в анкете на входе в кинотеатр vs
обучаем нейросеть распознавать настроение посетителя по выражению
лица. Первое — простое и предсказуемое, второе — сложное и точное.
Для старта мы выбираем галочки.

Почему мы пока не идём в ML:

- Нет размеченных данных. Чтобы обучить классификатор «хороший /
  плохой ответ», нужны тысячи примеров с разметкой. У нас их нет
  и не появится до запуска feedback-виджета.
- **NLP** (natural language processing — методы понимания смысла
  текста машиной) для оценки faithfulness стоит лишний LLM-вызов
  на каждый ответ. Это +latency и +деньги без подтверждённой
  пользы.
- Структурная эвристика покрывает «90% case'а»: ответ без citations
  всегда подозрителен, ответ с распределёнными — почти всегда
  хорош. Тонкие случаи решит ML потом.

Когда соберём golden-набор feedback'а — вернёмся и сравним нашу
эвристику с FactScore-style оценкой.

### 2. Hallucinated citations не считаются

Парсер сравнивает каждый найденный work_id со списком `sources`
из ответа и оставляет только реально присутствующие. Если модель
написала `[mn99]`, а в sources такого work_id нет — это галлюцинация.

Можно было бы наоборот, считать «модель уверена раз сослалась» —
но это поощряет именно то поведение, от которого мы защищаемся.
Сослался на несуществующее = понизили tier, не повысили.

### 3. Distribution check, не только counting

Можно было бы считать только число citations и игнорировать их
позиции. Но это давало бы зелёный tier ответам типа «согласно
[mn10], [sn56.11], [dn22]… дальше три абзаца от себя». Frontload —
плохой сигнал, и игнорировать его нельзя.

**Faithfulness** (RAG-термин — верность ответа найденным
источникам) выше у ответов, где ссылки **продолжают появляться**
к концу. Это значит модель опирается на корпус всю дорогу, а не
только во вступлении.

Аналогия: well-grounded ответ — диссертация с цитатами на
каждом утверждении до самой последней страницы. Frontload — речь
на свадьбе, в которой только первая фраза «как сказал классик…»
а дальше тосты от души.

### 4. Цветовое семейство, не радуга

Все цвета взяты из четырёх семейств Tailwind: `emerald`, `amber`,
`orange`, `rose` — плюс `destructive` для пустых результатов.
Шкала плавная: зелёный → жёлтый → оранжевый → розово-красный →
красный. Скачка «зелёный, потом сразу синий» нет — глаз
интуитивно читает прогрессию опасности.

В UI используем три согласованных оттенка одного семейства:
border, background, text. Через модификатор `dark:` подменяем
на более яркие версии для тёмной темы. Это типичный паттерн
shadcn/ui-стиля.

### 5. ARIA для accessibility

Бейдж — не только визуальный элемент. Слепым пользователям с
**screen reader'ами** (программами, читающими содержимое экрана
голосом) тоже надо понять уровень доверия.

Используем два **ARIA-атрибута** (Accessible Rich Internet
Applications — стандарт W3C для подсказок ассистивным технологиям):

- `role="status"` — атрибут, говорящий screen reader'у «это
  статусное сообщение, прочитай его когда обновится». Reader
  озвучит изменение бейджа без перебивания текущего чтения.
- `aria-label="Confidence: well-grounded"` — текстовая подпись
  элемента специально для screen reader'ов. Без неё reader
  прочитал бы только содержимое — `WELL-GROUNDED — 3 sources
  cited` — но не понял бы, что это **уровень доверия**, а не
  заголовок секции.

Аналогия: `role="status"` — это «нашивка» на одежде, которая
говорит стиральной машине «бережный режим». Атрибут невидим
визуально, но критичен для правильного обращения.

### 6. useMemo бесплатно, но используем

Вычисление `computeConfidence` дешёвое — линейный проход по
строке регуляркой, обычно меньше миллисекунды. Без `useMemo`
функция бы пересчитывалась на каждый рендер, и это было бы
не страшно.

Но `useMemo` тут стоит почти ничего, а гарантирует стабильную
ссылку на возвращаемый объект — это важно, если позже мы захотим
передавать результат в дочерний компонент с `React.memo`.
Дешёвая страховка.

## Что НЕ делаем

| Фича | Куда переехало |
|---|---|
| Info-tooltip с описанием каждого tier'а при наведении | app-day-26+ — добавим вместе с feedback-виджетом |
| ML-классификатор faithfulness в стиле FactScore | долгосрочно — после набора golden-данных через 👍/👎 |
| Per-style thresholds (для `concise`-стиля можно простить меньше citations) | app-day-26+ — после калибровки на реальных данных |
| Численный score 0.0-1.0 вместо tier'ов | не делаем — score типа 0.92 («оценка как 4.6 из 5 звёзд») создаёт ложное впечатление точности; tier'ы честнее показывают грубость метода |
| Учёт длины sources / overlap текста ответа с chunk-content'ом | потом, после feedback-калибровки |
| Confidence на streaming (live update во время печати) | app-day-25 streaming делает ответ кусками — пока ждём `done`-event и считаем один раз в конце |

## Как проверить

В stub-режиме (когда RAG возвращает фиксированные данные —
удобно для frontend-разработки без OpenRouter):

```
cd C:\Users\PChia\Dharma-RAG; pnpm --filter web dev
```

Открыть `http://localhost:3001/chat`, отправить любой запрос.
Под latency-строкой должен появиться зелёный бейдж
`WELL-GROUNDED` с пояснением «3 sources cited and references
are spread through the answer». Stub всегда возвращает 3 source'а
и ответ с распределёнными citations — это стартовый sanity-check.

Чтобы увидеть **другие tier'ы**, нужны реальный режим и разные
запросы. В реальном режиме (DeepSeek V4 Flash через OpenRouter):

Запрос про джхану — обычно 3-4 уникальные citations распределённо,
ожидаем `well-grounded`:

```
curl.exe -X POST http://localhost:8000/api/answer -H "Content-Type: application/json" -d '{"query":"что такое первая джхана?","top_k":5}'
```

В ответе считаем уникальные `[xxxNN]` в `answer` и смотрим где
последняя — должна быть после 60% длины. В UI бейдж зелёный.

Запрос вне корпуса — модель должна отказаться отвечать,
ожидаем `no-sources`:

```
curl.exe -X POST http://localhost:8000/api/answer -H "Content-Type: application/json" -d '{"query":"какой курс bitcoin сегодня?","top_k":5}'
```

В UI бейдж красный, label `NO SOURCES`.

Borderline-запрос с одним релевантным текстом — ожидаем
`limited grounding` (оранжевый):

```
curl.exe -X POST http://localhost:8000/api/answer -H "Content-Type: application/json" -d '{"query":"что такое sotapatti?","top_k":3,"forbidden_works":["mn10","sn56.11"]}'
```

Параметр `forbidden_works` — список work_id, которые retrieval
пропустит. Через него мы искусственно сужаем базу до 1-2 работ,
чтобы получить пограничные tier'ы для UI-проверки.

Browser-тест — DevTools → элемент бейджа должен иметь
`role="status"` и `aria-label="Confidence: <label>"`. В Lighthouse
accessibility-аудит не должен ругаться на отсутствие подписи.
