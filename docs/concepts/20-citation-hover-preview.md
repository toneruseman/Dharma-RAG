# 20 — Hover-preview для citations (app-day-23)

> **Статус:** реализовано в app-day-23. Малый chat-polish инкремент:
> наводишь курсор на бейдж `[mn10]` — выскакивает мини-карточка
> со snippet'ом (короткий фрагмент 1-2 предложения) из соответствующего
> источника.

## Зачем

После app-day-22 chat работает: пользователь задаёт вопрос, видит ответ
со ссылками-сносками вида `[mn10]` или `[sn56.11]`. Эти бейджи —
кликабельные **citation'ы** (ссылки на источник, как сноска внизу
страницы в книге). Клик ведёт в **Reading Room** (отдельная страница
чтения дхарма-текста по адресу `/read/{work_id}#{segment_id}`).

Проблема: чтобы понять «о чём вообще цитата `[mn10]`», нужно кликнуть,
дождаться загрузки Reading Room, прочитать кусок и вернуться обратно
в чат. Если в ответе пять citation'ов и пользователь хочет понять
куда углубиться — это пять кликов туда-сюда.

Аналогия: как если бы в Википедии каждая сноска вела сразу на отдельную
страницу без preview — ты бы потерял нить чтения. На самом деле там
hover-карточка показывает превью статьи, и ты решаешь стоит ли уходить.

Hover-preview даёт тот же **быстрый peek** в чате: задерживаешь курсор
на бейдже — через ~150мс выскакивает popover (всплывающая мини-панель
с информацией) со snippet'ом и score'ом (нормированный 0..1 показатель
релевантности — насколько этот источник подходит к запросу). Понимаешь
суть, не уходя из чата. Особенно полезно при multi-citation `[mn39, dn10]`:
наводишь поочерёдно, выбираешь куда углубиться по существу.

## Что такое Tooltip / popover / hover-preview — простыми словами

Три близких термина, которые часто путают:

- **hover** — событие, когда курсор мыши задерживается над элементом
  (не клик, не пролёт мимо — именно «остановился и стоит»).
- **popover** — всплывающая мини-панель, которая появляется рядом
  с элементом и показывает дополнительную информацию. Может содержать
  что угодно: текст, картинки, кнопки, целые формы.
- **tooltip** — частный случай popover: короткая всплывающая подсказка,
  обычно одна строка-две. «Подсказка при наведении» по-русски.

Аналогия: popover — это всплывающая мини-карточка со spoiler-текстом.
Наводишь — видишь что внутри. Tooltip — её самый простой вариант
(одна фраза-подсказка), а в нашем случае tooltip богаче: заголовок
с метаданными плюс snippet текста.

В проекте мы используем компоненты `Tooltip`, `TooltipTrigger`,
`TooltipContent` из библиотеки **shadcn/ui** (популярная коллекция
готовых React-компонентов: ты копируешь их код к себе в проект и
дальше владеешь им сам, в отличие от npm-зависимости). shadcn/ui
сам построен поверх **@base-ui/react** — это нижний слой, безголовая
(headless) UI-библиотека, которая занимается логикой и доступностью,
но не стилями. Стили на shadcn — Tailwind. У @base-ui свой API,
который местами отличается от старого `radix-ui` (предыдущая
headless-библиотека shadcn'а), и это будет важно дальше.

## Архитектура

```
AnswerView (рендерит ответ LLM с citation-бейджами)
  │
  │  для каждого работ_id в response.sources:
  │      Map<work_id → Source>  (быстрый lookup за O(1))
  │
  └─ <CitationBadge workId={"mn10"} source={highestScoreSource} />   ← новый компонент
       │
       ├─ <Tooltip>
       │   ├─ <TooltipTrigger render={<Link>}>     ← бейдж как ссылка
       │   └─ <TooltipContent>                     ← всплывающая мини-карточка
       │       ├─ заголовок: work_id · segment_id    score
       │       └─ тело: snippet (.dharma-text)
       │   </Tooltip>
```

Что здесь происходит на пальцах:

1. `AnswerView` получает `response` от backend'а — там есть `sources`
   (массив найденных фрагментов) и текст ответа со встроенными `[mn10]`.
2. Один раз строим **Map<work_id, Source>** — хеш-таблица, которая по
   `work_canonical_id` (стабильный идентификатор работы — например
   `mn10` для Majjhima Nikaya 10) быстро находит объект источника.
   Аналогия: оглавление в книге по номеру главы — даёшь номер, сразу
   видишь страницу. Без Map'а пришлось бы каждый раз пробегать массив
   `sources` циклом — медленно и многословно.
3. Для каждого встречного `[mn10]` в тексте парсер citations создаёт
   `<CitationBadge workId="mn10" source={…} />`.
4. `CitationBadge` рисует ссылку (`<Link>` ведёт в Reading Room) и
   оборачивает её в `Tooltip`. По hover'у `<TooltipContent>` всплывает
   с заголовком и snippet'ом.

`<TooltipProvider delay={150}>` уже глобально подключён в
`web/app/layout.tsx` (app-day-04). **delay** — задержка в миллисекундах
перед появлением tooltip'а: ставит «ты намеренно навёл, а не просто
проходил мимо». 150мс — компромисс: меньше = спам всплывашек при
скольжении мышью, больше = ощущается тормозным. Локальный provider
оборачивать `CitationBadge` не нужно — глобальный уже всё делает.

## Ключевые решения

### 1. Map выбирает highest-score source, а не первый попавшийся

В RAG-выдаче часто бывает: одна работа представлена **несколькими
segment'ами** (segment — независимо адресуемый кусочек работы, например
один параграф или одна шлока). Запрос «mindfulness» может вернуть
top-5 = `mn10:8.1`, `mn10:12.3`, `mn10:46.1`, `sn56.11:2`, `dn22:5` —
три разных segment'а одной MN10 плюс два других источника.

Когда пользователь видит в ответе `[mn10]` (без segment id), куда
он должен попасть по клику и какой snippet надо показать в hover'е?
Решение: **тот segment, у которого максимальный score** — он же
куда уйдёт пользователь по клику. Логика и в hover'е, и в клик-навигации
должна совпадать, иначе пользователь увидит одно в preview, кликнет,
попадёт совсем в другое место — рассинхрон.

```typescript
const sourceByWorkId = useMemo(() => {
  const map = new Map<string, Source>();
  for (const source of response.sources) {
    const existing = map.get(source.work_canonical_id);
    if (!existing || source.score > existing.score) {
      map.set(source.work_canonical_id, source);
    }
  }
  return map;
}, [response.sources]);
```

Прозой: пробегаемся по всем `sources`, для каждого `work_canonical_id`
держим в Map только тот, у кого score больше. Если для `mn10` уже
лежит segment со score 0.85, а пришёл с 0.92 — заменяем. На выходе
Map: `mn10 → segment-с-score-0.92`, `sn56.11 → ...` и так далее.
score 0.92 в этом контексте — оценка релевантности «как 4.6 из 5
звёзд для отзывов»: близко к идеалу, но не максимум.

**`useMemo`** — React-хук кеширования вычислений. Аналогия: закладка
с готовым результатом. Мы говорим «вот функция, вот зависимости
(`[response.sources]`) — пересчитай только если зависимости поменялись,
иначе верни прошлый результат». Без `useMemo` Map бы пересоздавался
на каждый рендер `AnswerView` (а рендеров много: каждый клик в UI
триггерит rerender). С ним — пересчёт только когда пришёл новый
ответ от backend'а.

### 2. `render` prop вместо `asChild` — чтобы Trigger выглядел как Link

Тонкий момент API. По умолчанию `<TooltipTrigger>` рендерит свой
обёрточный `<button>`. Но мы хотим, чтобы триггером был **наш
компонент** — `<Link>` из Next.js (он умеет client-side навигацию
без перезагрузки страницы). Как сказать «эй, не рендери своё
поведение, используй вот этот элемент»?

Два паттерна решения:

- **render prop** (паттерн: компонент принимает в свойстве другой
  компонент или JSX как «то, чем себя нарисовать»). В @base-ui это
  выглядит так:

  ```tsx
  <TooltipTrigger render={<Link href={target}>{workId}</Link>} />
  ```

  Аналогия: «нарисуй себя ВОТ этим элементом, а не как у тебя по
  умолчанию». TooltipTrigger всю логику hover'а/клавиатуры/ARIA
  навешивает на переданный `<Link>`.

- **asChild prop** — radix-аналог того же паттерна, другой способ
  передать кастомный элемент. Выглядит так:

  ```tsx
  <TooltipTrigger asChild><Link href={target}>{workId}</Link></TooltipTrigger>
  ```

  Семантически то же самое («не рендери свою кнопку, склей props с
  единственным child'ом»). Но это **radix-ui API**, а мы перешли на
  **@base-ui/react**.

Мы используем `render` prop, потому что shadcn/ui мигрировал на
@base-ui. Если копи-пастить старые примеры из интернета с `asChild`,
будет либо ругаться TypeScript, либо tooltip просто не сработает
молча. На этой граблине легко наступить — поэтому фиксируем явно.

### 3. Graceful fallback на missing source

Что делать, если в Map нет source'а для данного `workId`? Это
**hallucinated citation** — выдуманный work_id, которого нет в
`response.sources`. LLM иногда галлюцинирует: пишет в ответ `[mn99]`
просто потому что «звучит похоже на mn10». Парсер в
`web/lib/citations.ts` такие выдумки фильтрует ещё до рендера, но
защититься на уровне компонента полезно — defensive programming.

```typescript
if (!source) {
  return link;  // plain Link без Tooltip-обёртки
}
```

Прозой: если Map'а вдруг нет матча, возвращаем простой `<Link>`
без tooltip-обёртки. Бейдж остаётся кликабельным, ведёт в Reading
Room (где работа существует — иначе 404 и это уже отдельная история),
просто без preview-карточки. Никаких exception'ов в консоль, никаких
сломанных React-деревьев.

### 4. Tooltip-content повторяет layout SourcesPanel — для consistency

В правом боковом `SourcesPanel` (карточки sources, app-day-19/20)
каждая карточка показывает: сверху строка `work_id · segment_id`
слева и `score` справа, ниже — snippet в шрифте `.dharma-text`
(Noto Serif — серифный шрифт для дхарма-текстов, объявлен в
`17-base-layout.md`).

В hover-preview мы делаем тот же layout. Зачем: пользователь видит
два места, где живут источники (боковая панель и hover-карточка) —
если они выглядят по-разному, мозг тратит лишние циклы на
«это другая информация или та же?». Одинаковый layout = «ага,
это превью того же что в панели, понятно».

Размер ограничен утилитой Tailwind **`max-w-sm`** (max-width small ≈
384px). **Tailwind** — CSS-фреймворк где стили применяются классами-
утилитами вроде `max-w-sm`, `text-xs`, `whitespace-normal` прямо в
JSX, без отдельного CSS-файла. 384px достаточно для большинства
snippet'ов в одну-две строки. Для длинных подключаем
**`whitespace-normal`** — CSS-правило «разрешить перенос строк в
этом блоке» (по умолчанию в некоторых контекстах текст в одну строку
не переносится и вылезает за границы). С `whitespace-normal`
длинный snippet корректно обернётся внутри 384px.

ARIA-атрибуты (Accessible Rich Internet Applications — стандартные
HTML-атрибуты для screen-reader'ов; озвучивают что это за элемент
слепым пользователям) добавлены автоматически @base-ui'ем. На бейдж
`<Link>` навешивается `aria-describedby`, который указывает на
content tooltip'а — screen reader сначала прочитает текст ссылки,
потом «описание: snippet текст». Ничего вручную писать не пришлось,
библиотека делает это сама — одна из причин, почему мы взяли
готовый Tooltip, а не лепили свой div.

## Как проверить

Команды одной строкой (PowerShell вставляет только первую строку
из multi-line блоков).

Запустить backend в stub-режиме (RAG_BACKEND=stub отдаёт фиксированные
ответы без обращения к LLM/Qdrant — для UI-теста этого достаточно):

```
cd C:\Users\PChia\Dharma-RAG; .\.venv\Scripts\activate.ps1; $env:RAG_BACKEND="stub"; uvicorn src.api.app:app --reload --port 8000
```

В отдельном окне — frontend:

```
cd C:\Users\PChia\Dharma-RAG; pnpm --filter web dev
```

Открой `http://localhost:3001/chat`, отправь любой запрос (например
«what is mindfulness?»). В ответе будут citation-бейджи (`[mn10]`,
`[sn56.11]`, `[dn22]` от stub'а).

Что проверяем по пунктам:

1. **Hover на бейдж.** Наведи курсор на `[mn10]` и **задержи** на
   ~200мс. Через ~150мс должен появиться tooltip:
   - заголовок (моноширинный шрифт): `mn10 · mn10:8.1` слева,
     `0.92` справа
   - тело: snippet («Just mindful, they breathe in...» в stub'е),
     серифный шрифт `.dharma-text`
2. **Уход курсора.** Убери курсор с бейджа — tooltip исчезнет
   через ~150мс (тот же `delay` работает в обе стороны).
3. **Multi-citation.** Если в ответе есть `[mn10, sn56.11]`,
   наводись на каждый бейдж по очереди — должен появляться **разный**
   tooltip (с разным snippet'ом и score'ом). Если оба показывают
   одно и то же — баг в Map-логике.
4. **Клик по бейджу.** Кликни на `[mn10]` — должен уйти на
   `/read/mn10#mn10:8.1` (тот же segment, что показан в hover —
   highest-score). Если в URL другой segment_id — рассинхрон Map'а
   и навигации.
5. **Keyboard.** Кликни в textarea, нажми Tab несколько раз — фокус
   должен пройти по citation-ссылкам. На сфокусированной — tooltip
   появляется так же как при hover'е (это поведение по умолчанию
   @base-ui для accessibility).
