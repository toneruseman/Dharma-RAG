# 14 — Палийский глоссарий и query-expansion (rag-day-23)

> **Статус:** реализовано в rag-day-23. Default — `glossary_expand_pali_default=False`,
> флипается если mini-eval (`docs/EVAL_PALI_GLOSSARY.md`) показывает положительный
> сигнал на n=100 golden v0.0-extended.

## Зачем

BGE-M3 — multilingual encoder, и он неплохо стягивает английский ↔ русский.
Но у него есть **слепое пятно на бесконтекстных палийских словах** и на
**кириллических транслитерациях**:

- Корпус сейчас на ~99% английский (после rag-day-04). В нём слова в
  формах `jhāna`, `dukkha`, `meditative absorption`, `suffering`.
- Пользователь спрашивает «что такое **джхана**?» — это написание
  отсутствует в корпусе и плохо стягивается эмбеддингом к нужным чанкам.
- Пользователь спрашивает «что такое **jhana**» (без диакритик) — тоже
  не идеально совпадает с `jhāna`.

**Решение:** перед encode'ом расширять текст запроса канонической
палийской леммой плюс её EN/RU значениями. Запрос превращается из

> «что такое джхана?»

в

> «что такое джхана? jhāna meditative absorption медитативное погружение медитация»

Эмбеддер видит расширенную строку и тянет нужные чанки.

## Источники данных

Два слоя:

### 1. DPD (Digital Pāḷi Dictionary)

- 50 060 уникальных палийских лемм
- 100% покрытие английскими переводами, ~99.99% русскими
- Источник: [github.com/sc-voice/ms-dpd](https://github.com/sc-voice/ms-dpd)
  — это сжатая JS-версия полного DPD (8 MB вместо 2 GB SQLite)
- Лицензия: **CC-BY-NC 4.0** (см. ниже секцию о лицензии)
- Данные хранятся в `data/glossary/dpd_full.json` (10 MB), генерируются
  скриптом `scripts/build_dpd_glossary.py` из 4 `.mjs`-файлов ms-dpd
  (raw `.mjs` в git **не коммитятся** — `.gitignore`)

### 2. Кириллический ручной слой

- 155 high-frequency терминов с 288 вариантами кириллических написаний
- Покрывает то, чего у DPD нет: **палийские слова кириллицей**
  («джхана», «дуккха», «нирвана») и **санскритские формы** в принятом
  русском буддийском узусе («карма» вместо «камма», «дхарма» вместо
  «дхамма»)
- Файл: `data/glossary/cyrillic.yaml` (рукой написан, ревьюится как код)
- Источники для подбора: Ньянатилока «Словарь буддийских терминов»
  (русский перевод), theravada.ru / dhamma.ru глоссарии, устоявшийся
  узус в русскоязычной буддийской литературе

В Phase 3 ручной слой планируется расширить до 500-2000 терминов.

## Алгоритм expand_query

```
expand_query("что такое джхана?") -> ?
```

1. **Токенизация.** Простой regex по unicode-letter-runs, дефисы внутри
   слов сохраняются (для составных типа «благородная-истина»).
   Результат: `["что", "такое", "джхана"]`.

2. **Lookup каждого токена** — два пути, разные правила входа:
   - **Кириллица всегда разрешена.** Если токен в `cyrillic.yaml`
     (`джхана`, `карма`) — ищем по нему. Кириллица не пересекается
     с английским, шума не даёт.
   - **Pāli — только с диакритиками.** Если токен содержит хотя бы
     один combining mark (`jhāna`, `paṭiccasamuppāda`) — ищем в DPD
     напрямую, и через strip-diacritics на случай нестандартной
     нормализации. Если нет диакритик (`buddha`, `dukkha`, `sutta`)
     — **скипаем**, даже если эта lemma есть в DPD.

   **Зачем такой строгий guard на ASCII:** day-23 mini-eval показал,
   что DPD содержит ~50k лемм, включая односимвольные служебные
   слова (`a`, `na`, `va`) и обычные английские заимствования
   (`buddha`, `sutta`, `kamma`). На английских запросах эти леммы
   шумят: расширение `Buddha` в `Awakened One Будда Пробужденный`
   ломает semantic context для английского retrieval, потому что
   корпус и так понимает английское "Buddha" в нужном смысле.
   Реальная польза глоссария — там, где у encoder'а пробел: bare-Pāli
   с диакритиками + кириллические транслитерации. См. секцию
   «Результаты mini-eval» ниже.

3. **Расширение.** Для каждого распознанного термина:
   - добавляем каноническую палийскую лемму (если ещё не в запросе)
   - добавляем top-2 EN значений
   - добавляем top-2 RU значений
   - дедупликация против оригинальных токенов (если «suffering» уже
     был в запросе — не дублируем)

4. **Лимиты.**
   - `max_terms=5` — максимум 5 распознанных терминов на запрос
     (защита от раздувания на полностью палийских предложениях)
   - `max_meanings=2` — максимум 2 значения на язык на термин

5. **Возврат.** Если ни один токен не распознался — возвращаем исходный
   запрос без изменений.

## Интеграция в pipeline

```
QueryRequest -> RAGService.query
                ├─ resolve effective expand_pali (request override > setting)
                ├─ if True and glossary loaded:
                │     query_text = glossary.expand_query(request.query)
                ├─ encode(query_text)
                ├─ hybrid_search(...)
                └─ response: query=request.query (исходный),
                             metadata.expand_pali=<effective>
```

Несколько важных свойств:

- **Расширение происходит ДО encode'а.** Эмбеддер видит расширенную
  строку, в Qdrant уходит вектор от неё.
- **Ответ возвращает исходный `request.query`.** Пользователь ничего
  не «теряет» — расширение это внутренняя оптимизация retrieval.
- **`metadata.expand_pali` — эффективный**, не запрошенный. Если
  `expand_pali=True` в запросе но ни один токен не распознан —
  metadata честно покажет `False`.
- **Stub backend (`RAG_BACKEND=stub`)** игнорирует expand_pali — ему
  всё равно, он отдаёт фикстуры.

## Контроль через API и окружение

```python
# По умолчанию (server-side):
glossary_expand_pali_default = False  # пока mini-eval не подтвердил win

# Явно через POST /api/query:
{"query": "...", "expand_pali": true}   # форсировать включение
{"query": "...", "expand_pali": false}  # форсировать отключение
{"query": "...", "expand_pali": null}   # default (None) — использовать server-side
```

Если файлы глоссария отсутствуют (например, в минимальном CI-образе),
factory логирует warning и инжектит `glossary=None` — запросы продолжают
работать, просто без expansion.

## Результаты mini-eval (rag-day-23)

`scripts/eval_pali_glossary.py` прогнан на synthetic golden v0.0-extended
(n=100, 91 EN / 7 RU / 2 pli) с production cell `dharma_v2 + rerank=False
+ expand=True`. Полный отчёт: [`docs/EVAL_PALI_GLOSSARY.md`](../EVAL_PALI_GLOSSARY.md).

### Прогон 1 — без diacritic-guard (первая итерация)

Расширяли любой токен совпавший с DPD-леммой, включая ASCII.

| metric | baseline | candidate | Δ |
|---|---:|---:|---:|
| ref_hit@1 | 0.190 | 0.230 | **+4.0 pp** |
| ref_hit@5 | 0.450 | 0.420 | **−3.0 pp** |
| MRR | 0.308 | 0.324 | +0.016 |
| fixed | — | — | 2 |
| regressed | — | — | **5** |

Анализ failure-логов выявил два класса шума:
1. Односимвольные палийские служебные слова (`a`, `na`) матчили
   английские `a`, `not`, `from` — расширение тащило `(гр) буква а
   первая буква алфавита` в каждый запрос.
2. Обычные заимствования (`Buddha`, `sutta`) — DPD добавлял
   «Awakened One Будда Пробужденный» в **каждый** английский запрос
   про Будду, размывая semantic context.

### Прогон 2 — с diacritic-guard

После добавления `_has_pali_diacritics` фильтра (см. выше):

| metric | baseline | candidate | Δ |
|---|---:|---:|---:|
| ref_hit@1 | 0.190 | 0.200 | +1.0 pp |
| ref_hit@5 | 0.450 | 0.450 | **±0** |
| ref_hit@10 | 0.540 | 0.530 | −1.0 pp |
| ref_hit@20 | 0.650 | 0.660 | +1.0 pp |
| MRR | 0.308 | 0.313 | +0.005 |
| fixed | — | — | 1 |
| regressed | — | — | 1 |

Шум устранён. По языкам:
- **EN** (n=91): `ref_hit@5` 0.473 → 0.473 (диакритик-guard блокирует
  расширение почти всех английских запросов — что и нужно)
- **RU** (n=7): `ref_hit@20` 0.286 → 0.429 (**+14 pp** — единственный
  явный сигнал, на крошечной выборке)
- **pli** (n=2): `ref_hit@10/20` 0.500 → 0.000 (одна из двух pālī
  QA регрессировала — выборка слишком маленькая, чтобы обобщать)

### Решение

`glossary_expand_pali_default = False` — не флипаем default. Сигнал на
n=100 слабый и в пределах шума. Код shipped и протестирован, доступен
через явный `expand_pali=True` в API-запросе или через env-флаг для
A/B экспериментов.

**Что меняет картину:**

1. **Phase 3 RU-rich golden v0.1** (планируется при появлении русского
   корпуса) — там доля cyrillic-запросов вырастет с 7/100 до 30-50/100,
   и эффект станет статистически значимым.
2. **Multi-word lookup** — сейчас «четыре благородные истины» не
   матчится. Добавление n-gram lookup может дать заметный лифт на
   фразовых запросах.
3. **Stop-list для общих английских заимствований** — даже с diacritic-
   guard, `paññā` (с диакритиками) расширяется в «understanding wisdom»
   — это может пересекаться с английским «wisdom» в корпусе.

## Ограничения и compromise'ы

### 1. Ambiguity по смыслу (sense disambiguation)

Слово `sutta` — и «scripture» (буддийский смысл, в нашем корпусе) и
«asleep» (общий смысл, причастие). DPD кладёт оба смысла в одну лемму
после нашей дедупликации, и top-2 значения могут отдать «не тот»
смысл первым.

Митигация: trust the encoder. Расширение `sutta sleeping asleep
лежащий спящий scripture` всё равно ближе к буддийскому контексту,
чем чистое `sutta`, благодаря semantic context из остальной части
запроса. Mini-eval покажет, насколько это работает на практике.

### 2. Полисемия и шум

50k DPD-лемм включают служебные слова, частицы, односложные грамматики.
Часть из них может случайно совпасть с не-палийскими словами в запросе
и добавить шум (`a`, `na`, `va` — это и palийские частицы тоже).

Митигация: `max_terms=5` режет blast radius. Если эмпирически проблема —
добавим стоп-лист в Phase 3.

### 3. Многословные термины

«Четыре благородные истины» / «Noble Eightfold Path» — это
**фразы**, не один токен. Текущая токенизация разделит их и не найдёт.

Митигация для будущего: n-gram lookup (2-3 токена подряд проверять как
ключ). Не сделано в rag-day-23 — добавится отдельной итерацией если
mini-eval покажет, что это узкое место.

### 4. Нестандартные транслитерации

Pāli `jhāna` пишется по-русски тремя способами: «джхана», «джана»,
«дхьяна». Мы стараемся покрыть все варианты в `cyrillic.yaml`, но новые
формы будут всплывать. Каждый месяц пройтись по логам непокрытых
кириллических токенов и пополнять — отдельная задача.

## Лицензия DPD: CC-BY-NC 4.0

DPD распространяется под **Creative Commons Attribution-NonCommercial
4.0**. Это означает:

- ✅ **Можно**: использовать в исследовании, для MVP, в open-source
  проекте без коммерциализации. Достаточно атрибуции.
- ❌ **Нельзя**: продавать сервис, который зависит от DPD; брать деньги
  за подписку на API, использующий DPD-расширение.

Перед коммерциализацией Dharma-RAG нужно либо:
1. **Договориться с авторами DPD** (`bodhirasa@digitalpalidictionary.com`)
   о коммерческой лицензии.
2. **Заменить DPD на open-license словарь.** Кандидаты: Pali Text Society
   editions (public domain), Cone «Dictionary of Pāli» (Creative Commons
   forms unclear), собственный словарь по Канону.
3. **Сделать expansion опциональной фичей**, которую отключают в платных
   тарифах.

В коде глоссарий грузится опционально (`factory._try_load_glossary`
возвращает None если файл отсутствует) — это даёт runway для смены
источника без переписывания integration.
