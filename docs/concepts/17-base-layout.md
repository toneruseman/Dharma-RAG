# 17 — Базовый layout web/ (app-day-04)

> **Статус:** реализовано в app-day-04. У `web/` появился theme-провайдер,
> shell-компоненты (Header / Footer), типографика под палийскую
> диакритику и landing-page со ссылками на главные surface'ы. Сам
> Reading Room (страница для долгого чтения текстов) — дело Phase 2
> (app-day-21+). Здесь только «строительные леса».

## Зачем

После app-day-03 у нас был type-safe API-клиент (TypeScript-функции,
которые знают форму запросов и ответов backend'а), но **web-приложение
оставалось default-stub'ом** — той самой стартовой страницей Next.js
со ссылками на шаблоны Vercel. Прежде чем строить Reading Room,
Search и Chat, нужен общий каркас, который будет на всех страницах.

**Что значит «общий каркас»?** В вебе это называется **layout**
(шапка с навигацией сверху, основное содержимое посередине, подвал
снизу — те части, которые повторяются на каждой странице сайта).

Аналогия: layout — это **каркас дома, в который въезжают разные
комнаты-страницы**. Стены, крыша, проводка — общие. А внутри одной
комнаты Reading Room с книгами, в другой — Chat с диалогом, в третьей
— Search с формой поиска. Без каркаса каждый раз пришлось бы заново
строить стены.

Конкретно нам нужны:

- единые `<Header/>` (шапка) и `<Footer/>` (подвал) на всех страницах
- light/dark тема (светлая и тёмная), переключающаяся без мерцания
- набор шрифтов с поддержкой палийской диакритики (`ā ī ū ṃ ṇ ñ ṭ ḍ ṅ ṣ ḷ`)
- landing-page (главная страница), объясняющий «что это и куда идти»

Это не фичи — это инфраструктура, чтобы следующие дни не упирались
в basics.

## Что такое layout / theme / typography — простыми словами

### Layout

**Layout** — общий каркас страницы: header (шапка) сверху, основное
содержимое в середине, footer (подвал) внизу. В Next.js
([React-фреймворк для веб-приложений; рендерит страницы и на сервере,
и в браузере, чтобы первый показ был быстрым](https://nextjs.org))
есть специальный файл `app/layout.tsx` — он автоматически оборачивает
**все** страницы. Написал шапку один раз — она появилась везде.

### Theme

**Theme** — это набор цветов (фоны, текст, рамки) и переменных,
который можно переключать одной кнопкой. У нас две темы: светлая и
тёмная. **Theme provider** (компонент-обёртка, которая «раздаёт»
настройку темы всем дочерним элементам) подключается на самом верху,
и любая кнопка / карточка / текст внутри узнаёт «сейчас тёмная или
светлая».

Аналогия: theme provider — **общий выключатель всех ламп в доме**.
Щёлкнул — и все лампы во всех комнатах сразу переключились. Не надо
ходить и щёлкать каждую отдельно.

### Typography

**Typography** — это всё, что про шрифты: какой шрифт где, какие
размеры, какие межстрочные интервалы. Для дхарма-текстов это особенно
важно, потому что в палийском языке много **диакритики** (надстрочные
и подстрочные знаки: `ā`, `ī`, `ū`, `ñ`, `ṭ` — без них слово
читается криво или вообще теряет смысл). Не каждый шрифт умеет их
рисовать красиво, и не каждый файл шрифта вообще содержит эти
символы.

## Архитектура

```
web/app/layout.tsx (RootLayout — корневой компонент)
  │
  ├─ <html lang="en" suppressHydrationWarning>
  │  │
  │  └─ <body class={...inter, serif, mono variables}>
  │     │
  │     └─ <ThemeProvider>           ← next-themes, раздаёт light/dark
  │        │
  │        └─ <TooltipProvider>      ← shadcn/ui, общая настройка тултипов
  │           │
  │           ├─ <Header />          ← sticky шапка с logo + nav + theme-toggle
  │           │
  │           ├─ {children}          ← сюда подставляется текущая страница
  │           │
  │           └─ <Footer />          ← disclaimer + ссылки
```

Что здесь происходит. Next.js при заходе на любую страницу сначала
рендерит `RootLayout`, потом внутрь него вставляет содержимое
конкретной страницы (`page.tsx`). Иерархия провайдеров идёт сверху
вниз: сначала тема (потому что от неё зависят цвета всего ниже),
потом тултипы (всплывающие подсказки), потом сам shell с Header и
Footer.

`{children}` — это «дырка», в которую Next.js вставляет текущую
страницу. На главной туда подставляется landing, на `/chat` —
страница чата, и т.д. Header и Footer при этом остаются на месте.

## Ключевые решения

### 1. Theme через next-themes + переключение CSS-only

Установлен `next-themes@0.4.6` — npm-пакет (готовая библиотека из
реестра npm), который умеет всё про темы: запоминает выбор в
`localStorage` (хранилище в браузере), следит за системной темой ОС,
и главное — корректно работает с **hydration**.

**Hydration** — после того как сервер прислал готовый HTML, в
браузере подгружается JavaScript и «оживляет» страницу: вешает
обработчики кликов, делает её интерактивной. Аналогия: «дом приехал
готовый с фабрики — стены, окна. Потом приходят электрики и
подключают свет, интернет, отопление». Hydration = «электрики
приехали».

Проблема с темами: если на сервере отрендерили светлую, а у юзера в
браузере сохранена тёмная — после hydration тема меняется. Юзер
видит **flash of unstyled content / FOUC** (мерцание неоформленного
контента: белый фон на полсекунды, потом резко тёмный). Уродливо.
`next-themes` это решает специальным inline-скриптом, который
выставляет правильный класс на `<html>` ещё **до** рендера body.

Конфиг провайдера в [`web/components/theme-provider.tsx`](../../web/components/theme-provider.tsx):

- `attribute="class"` — переключение через `<html class="dark">` (и
  Tailwind-варианты `dark:bg-black`, и наши CSS-переменные на
  `.dark` — все начинают работать)
- `defaultTheme="system"` — по умолчанию следуем за системной темой
  ОС (`prefers-color-scheme`)
- `enableSystem` + `disableTransitionOnChange` — без мерцаний при
  переключении

Кнопка-переключатель [`theme-toggle.tsx`](../../web/components/theme-toggle.tsx)
показывает иконки sun/moon из `lucide-react`. Хитрость: иконки не
переключаются через JS-условие (`{isDark ? <Moon/> : <Sun/>}`) — это
дало бы hydration-несоответствие. Вместо этого обе иконки **всегда
в DOM**, а CSS-классами `dark:opacity-0` одна прячется в светлой
теме, другая — в тёмной. Чисто CSS, никакого JS-условия — никакого
hydration-warning.

### 2. Шрифты через next/font/google

В [`web/app/layout.tsx`](../../web/app/layout.tsx) подключены три шрифта:

```tsx
import { Inter, Noto_Serif, JetBrains_Mono } from "next/font/google";

const inter = Inter({ subsets: ["latin"], variable: "--font-sans" });
const notoSerif = Noto_Serif({
  subsets: ["latin", "latin-ext", "vietnamese"],
  variable: "--font-serif",
});
const jetbrainsMono = JetBrains_Mono({ subsets: ["latin"], variable: "--font-mono" });
```

Что здесь происходит. `next/font/google` — это API Next.js, который
**на этапе билда** скачивает Google Fonts и кладёт в свою сборку.
Юзер не идёт за шрифтами на серверы Google в runtime — это и
быстрее, и приватнее (нет third-party request'а), и стабильнее (если
Google упал, наш сайт работает).

**Subsets** (подмножества шрифта — наборы символов). Полный файл
шрифта со всеми языками мира — мегабайты. Нам не нужны китайские
иероглифы и арабская вязь. Ставим только нужные subsets, файл
становится в разы меньше.

Аналогия: font subset — **алфавит специально для русского, без
коробки с китайскими иероглифами рядом**. На полке экономия места.

Для Noto Serif берём `latin` + `latin-ext` + `vietnamese`. Звучит
странно — при чём тут вьетнамский? Дело в том, что **palli-диакритика
(ā ī ū ñ ṭ) живёт в подмножестве `vietnamese`** в Google Fonts.
Эмпирический факт: эти три subset'а вместе дают полное покрытие
палийских символов. Я проверил — без `vietnamese` `ṭ` рендерится
как пустой квадратик.

**Зачем три шрифта:**

- **Inter** (`--font-sans`) — для UI (кнопки, меню, сообщения).
  Современный, читабельный, нейтральный.
- **Noto Serif** (`--font-serif`) — для дхарма-текстов в Reading
  Room. Серифы (засечки на буквах) делают длинные тексты комфортнее
  для чтения, плюс этот конкретный шрифт хорошо рисует диакритику.
- **JetBrains Mono** (`--font-mono`) — моноширинный, для кода и
  идентификаторов чанков (`mn10:7-12`).

`variable: "--font-sans"` — Next.js создаёт CSS-переменную, через
которую любой элемент может попросить этот шрифт без import'а.

### 3. Класс `.dharma-text` для красивой типографики

В [`web/app/globals.css`](../../web/app/globals.css):

```css
.dharma-text {
  font-family: var(--font-serif), Georgia, "Times New Roman", serif;
  font-feature-settings: "kern", "liga", "calt";
  line-height: 1.75;
  letter-spacing: 0.005em;
}
```

Этот класс — стиль для блоков длинного дхарма-текста (термин **prose
styling** = стили для длинного читабельного текста, в отличие от
коротких подписей и заголовков). Сейчас он только объявлен — мы его
ещё нигде не применяем. В Reading Room (Phase 2) появятся
`<div class="dharma-text">` вокруг сутт.

**`font-feature-settings`** включает три фичи OpenType-шрифтов:

- `kern` (kerning) — **расстояние между конкретными парами букв**.
  Например, `AV` в обычном рендере выглядит как `A V` с дыркой —
  kerning сдвигает их ближе, потому что наклон `A` и `V` оставляет
  визуальную пустоту. Без kerning'а текст читается «дёргано».
- `liga` (ligatures) — **специальные формы связки двух букв**.
  Классика: `fi` рисуется как один символ `ﬁ`, чтобы точка над `i`
  не врезалась в верх `f`. В палийском критично для сочетаний типа
  `ñj`, `ṭṭh`.
- `calt` (contextual alternates) — **альтернативные формы букв в
  зависимости от соседей**. Шрифт сам выбирает, какую глифу `h`
  нарисовать, если перед ней стоит `t` с точкой снизу. Без `calt`
  такие сочетания могут наезжать друг на друга.

Включаем все три, и Noto Serif начинает рисовать палийские слова
красиво вместо «свалки символов».

`line-height: 1.75` (большой межстрочный интервал) и
`letter-spacing: 0.005em` (чуть-чуть «дышать» между буквами) —
стандарт для long-form чтения. Глаза не устают на длинных пассажах.

### 4. Sticky header + backdrop-blur

[`Header.tsx`](../../web/components/layout/Header.tsx) — это
**sticky-header** (заголовок, прилипающий к верху при прокрутке —
содержимое страницы скроллится под ним, а сам он остаётся на месте).
В Tailwind ([утилитарный CSS — пишешь классы прямо в HTML, типа
`flex p-4 bg-blue-500`, без отдельных `.css`-файлов](https://tailwindcss.com))
это `sticky top-0`.

Аналогия: sticky-header — **магнитная полочка над прокручивающимся
текстом**. Текст уезжает наверх, полочка остаётся.

Плюс эффект **backdrop-blur** (размытие фона позади полупрозрачного
элемента). Header сам полупрозрачен (`bg-background/80`), и фон под
ним размывается (`backdrop-blur`). Когда скроллишь страницу, контент
проступает сквозь шапку «через матовое стекло». Современная
эстетика, и удобно — видно, есть ли ещё контент сверху.

Содержимое: logo слева, навигация (Read / Search / Chat) посередине,
theme-toggle справа.

### 5. shadcn/ui для базовых компонентов

Через `pnpm dlx shadcn@latest add button card sheet dialog tooltip
separator scroll-area --yes` добавили 6 новых компонентов в
`web/components/ui/`. `button.tsx` уже был с app-day-01.

**Что такое shadcn/ui.** Это **набор готовых компонентов** (кнопки,
карточки, диалоги, тултипы) — но не как обычная npm-библиотека. У
shadcn своя философия: вместо `npm install` он **копирует файлы
прямо в твой репозиторий**. Получаешь не зависимость, а исходник,
который можешь править как угодно.

Аналогия: shadcn/ui — **магазин готовой мебели IKEA**. Берёшь
понравившийся диван, привозишь домой, собираешь по инструкции — и
он твой. Хочешь перетянуть обивку? Делай. С обычной npm-библиотекой
было бы как «арендовать диван у поставщика — нельзя ни перекрасить,
ни ножки подпилить, иначе сломаешь договор».

В нашем layout'е используются:

- `Card` (карточка с рамкой) — на landing-page для трёх surface'ов
- `Tooltip` + `TooltipProvider` (всплывающая подсказка / контейнер
  настройки для всех тултипов на странице) — `TooltipProvider`
  стоит глобально в layout, чтобы любой `<Tooltip>` где угодно
  работал без локальной обёртки
- `Sheet`, `Dialog` — задел для будущих модалок и сайдбаров

Параметр `delay={150}` у `TooltipProvider` — задержка перед показом
тултипа в миллисекундах. У shadcn это `delay`, не `delayDuration`
как у Radix UI (другая библиотека) — внимание, под капотом shadcn
сейчас использует `@base-ui/react`, и API у него своё.

## Как проверить

Команды одной строкой (PowerShell на Windows вставляет только
первую строку из multi-line блоков, поэтому даём single-line).

Поднять оба сервера (web на 3001, api на 8000):

```
cd C:\Users\PChia\Dharma-RAG; pnpm dev
```

Открыть `http://localhost:3001` в браузере. Что должны увидеть:

- **landing-page** с hero-блоком и тремя карточками (Read / Search / Chat)
- **Header sticky сверху** — при скролле остаётся на месте, контент
  проступает через `backdrop-blur`
- **Footer снизу** — disclaimer + ссылки на Sources / Audit / Privacy / GitHub
- **Тема переключается кнопкой sun/moon в правом верхнем углу** —
  без мерцания / FOUC. Перезагрузи страницу (F5) — выбранная тема
  сохранилась
- **Все ссылки в nav и карточках кликабельны** — 404 на
  `/read | /search | /chat` пока ОК (страницы Phase 2-4)

Дополнительные проверки качества кода (single-line):

```
pnpm --filter web tsc --noEmit
```

Должно завершиться без ошибок. Это **TypeScript-чек** (типы во всех
файлах согласованы), `--noEmit` — не генерируй JS-файлы, только
проверь.

```
pnpm --filter web lint
```

ESLint-проверка ([линтер — статический анализатор кода, ловящий
плохие паттерны до запуска](https://eslint.org)) — должна быть
clean.

```
pnpm --filter web build
```

Production-билд (полная сборка для деплоя). Должен завершиться
зелёным. Если падает — что-то в layout сломано.
