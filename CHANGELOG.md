## [0.2.1] — 2026-05-09

### Added
- **rag-day-36:** Бесконечный тред — LLM-free passage rotation (concept 36). Флагманская фича Yoniso/Dharma-RAG: пользователь задаёт вопрос один раз, нажимает «Далее» → получает следующий канонический chunk из корпуса по тому же запросу, без LLM-вызова. Каждая карточка несёт `context_text` (pre-baked Contextual Retrieval prefix из rag-day-16) + сам текст + `(work, segment, translator, language, score)`. **Stateless backend**: клиент сам ведёт `excluded_chunk_ids[]` и шлёт массив в каждом раунде. **Стоимость per round: $0** (нет LLM-токенов; одна Qdrant + Postgres FTS + один JOIN). **Latency: ~200ms** (vs 14-25s для LLM-mode), что даёт мгновенный feel «листай далее». **Zero hallucination** — каждое слово verbatim из канона. **Backend:** новый `src/api/thread.py` router (`POST /api/thread/next`), новый `RAGService.thread_next()` метод (использует `expand_parents=False` чтобы карточка была bite-sized child ~384 токена, отдельный Postgres SELECT для `context_text`/`translator`/`language` enrichment), новые schemas `ThreadRequest`/`ThreadCard`/`ThreadResponse` в `src/rag/schemas.py`, расширен `RAGServiceProtocol`, добавлен `StubRAGService.thread_next` для frontend dev. **Frontend:** новая страница `/thread` (`web/app/thread/page.tsx`), новый компонент `<PassageCard>` с layout «бейдж раунда + work/segment + score → курсивный context_text intro → текст verbatim → footer translator/language + Reading Room link», nav-link «Thread» в Header. Регенерирован `openapi.json` + `lib/api-types.ts`; добавлены `threadNext()` async-функция и type-aliases в `lib/api-client.ts`. **Trade-offs (см. concept-36):** сейчас diversification только по `chunk_id` (один work может появиться много раз через разные children) — после ATI/Dharmaseed добавим фильтр по `(work, translator)`. Score-threshold autostop (вместо `len(filtered) < top_k`) — TODO. LLM/source toggle в `/chat` — отдельной задачей. **Никаких изменений в src/retrieval/** — переиспользуем `hybrid_search`. **Файлы:** новые `src/api/thread.py`, `web/app/thread/page.tsx`, `web/components/thread/PassageCard.tsx`, `docs/concepts/36-infinite-thread.md`; modified `src/rag/schemas.py`/`service.py`/`protocol.py`, `src/api/_rag_stub.py`/`app.py`, `web/lib/api-client.ts`, `web/components/layout/Header.tsx`, `web/app/chat/page.tsx` (добавлены 2 missing optional metadata fields после typegen-regen), regenerated `openapi.json`/`lib/api-types.ts`. 454/454 unit-тестов green. mypy + tsc + eslint clean.

The pipeline is functionally correct; latency optimization (batch tuning, fp16 verification, possibly ONNX runtime) is deferred to a follow-up. `rerank=false` gives the day-12 fast path (60-200 ms) for users who need throughput.

- **rag-day-12:** Hybrid retrieval via Reciprocal Rank Fusion (RRF, k=60) over dense BGE-M3 + sparse BGE-M3 + BM25 Postgres FTS. New modules: `src/retrieval/{rrf,dense,sparse,hybrid,schemas}.py`. `POST /api/retrieve` endpoint mounted on the FastAPI app — first user-facing search API; returns ranked chunks with per-channel ranks for observability. **Latency on GPU: 62-96 ms/query** end-to-end (encode 46-53 ms, channels 14-46 ms parallel, fusion <1 ms, enrich 1-11 ms) — well under the day-12 plan's 200 ms target. **35 new unit tests** (RRF formula, channel pass-through, orchestrator with fakes), 100% coverage of `rrf.py`, `dense.py`, `sparse.py`, `schemas.py`. Smoke script `scripts/smoke_hybrid.py` validates 10 canonical queries against the live stack.
- **Fusion concretely closes day-10/11 gaps:** for `satipaṭṭhāna` (where dense scored 0.49, sparse was weak, BM25 returned 0), hybrid surfaces AN 10.67 / AN 10.68 — passages where the Pāli term actually appears in Sujato's English. RRF rewarded the document for ranking 10/12 across two channels even though no single channel led with it.

### Removed
- **`docs/old/`** (30 files, ~16 K lines) — pre-ADR-0001 documentation: outdated parameters (150/600 word chunks, Langfuse-as-primary), superseded research (knowledge graphs vs vector search, etc.). All current direction lives in `docs/decisions/`, `docs/RAG_DEVELOPMENT_PLAN.md`, and the in-repo READMEs. Closes long-standing blocker B-002.

Pre-commit hooks (ruff, ruff-format, mypy, detect-secrets) keep the local safety net intact. Re-introduction tracked as B-004 (`v0.1.0 Foundation` milestone) using uv (Astral) instead of pip.

### Added
- **rag-day-11:** BM25-style lexical retrieval via Postgres FTS in `src/retrieval/bm25.py`. Alembic migration 003 adds `chunk.fts_vector` as a `GENERATED STORED` `tsvector` derived from `text_ascii_fold`, with a GIN index for O(log N) `@@` match. Uses the `simple` text-search config (no stemming) so Pāli terms retain full token identity and the client-side `normalize_query()` mirrors day-6's `to_ascii_fold` + lowercase — `satipaṭṭhāna` and `satipatthana` produce identical tsqueries. Uses `websearch_to_tsquery` for safe user-query parsing and `ts_rank_cd` for cover-density scoring (IDF-weighted, position-aware — close enough to BM25 for hybrid fusion). 9 unit tests (pure-function normaliser + dataclass contract) + 8 integration tests against live Postgres (term-density ranking, diacritic folding both directions, parent exclusion, limit respect). 100% coverage of `bm25.py`.
- **Honest corpus note:** Sujato's English translation renders most Pāli doctrinal terms into English ("Satipaṭṭhāna Sutta" → "Mindfulness Meditation"), so BM25 on the current corpus retrieves proper nouns (`Anāthapiṇḍika` → MN 143 at 0.80), place names (`Sāvatthī`), and English doctrinal terms (`four noble truths` → SN 56.*, `noble eightfold path` → SN 45.*), but returns zero hits for bare Pāli queries like `satipaṭṭhāna` / `anapanasati`. This gap is by-design for day-11's pure-lexical channel — day-12 hybrid fusion pairs it with dense BGE-M3 (which DOES handle paraphrased/cross-lingual queries), and day-16 contextual retrieval adds Pāli uid + title to each chunk's embedding input, closing the gap for all three channels.
- **rag-day-10:** Qdrant indexer in `src/embeddings/indexer.py`. Named-vector collection `dharma_v1` (1024-d dense BGE-M3 cosine + learned sparse). Pure DI via `QdrantClientProtocol` + `EncoderProtocol` so unit tests run without Qdrant or CUDA (19 unit tests, 98% coverage). Integration tests verify real qdrant-client 1.17 accepts the `PointStruct` shape, `query_points` round-trips dense/sparse, identity-vector query returns cosine 1.0, and re-upsert on the same chunk UUID is idempotent (5 integration tests). CLI `scripts/index_qdrant.py` streams from Postgres with keyset pagination on `chunk.id`, filters to child chunks by default (`--include-parents` to override), supports `--limit N` for smoke runs and `--recreate` to drop the collection. **Full corpus indexed on GPU: 6,478 child chunks in 4:40 min on the 1080 Ti with fp16 (~23 chunks/sec), 0 failed batches.** `scripts/smoke_retrieval.py` runs a curated set of canonical queries (English paraphrase, Pali with/without diacritics, Russian cross-lingual) and prints dense + sparse top-3 with scores and text — qualitative sanity gate before the numeric eval on day 14.
- **Toolchain:** torch upgraded 2.5.1+cu121 → 2.6.0+cu124 (CVE-2025-32434 + transformers 4.57 pickle-loader gate). huggingface-hub pinned back to `<1.0` to match transformers 4.x. Both were latent conflicts from day-8; day-10 forced the resolution.
