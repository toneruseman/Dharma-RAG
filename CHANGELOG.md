### Added
- **app-day-24:** Confidence indicator в чате. Под ответом — цветной бейдж с уровнем уверенности: `well-grounded` (зелёный) / `synthesized` (жёлтый) / `limited grounding` (оранжевый) / `interpretive — verify with teacher` (красный) / `no sources` (красный). Сигнал нужен потому что LLM-ответ выглядит одинаково уверенно при 0 и 5 citations — для буддийских текстов это опасно (interpretive ответ может звучать как доктрина, а быть домыслом). Industry-pattern (Glean / Perplexity / NotebookLM): visible confidence signal под ответом. **Логика структурная, без NLP** — пороги подбираются по числу уникальных citations + распределению по тексту. **Tier'ы:** `well-grounded` если ≥3 unique citations и последняя ≥60% от длины ответа (распределённо, не frontload); `synthesized` если ≥2 unique; `limited` если ровно 1; `interpretive` если 0 (модель говорит вне retrieved sources); `no-sources` если retrieval отдал 0 (модель отказалась). Hallucinated citations в счёт не идут — parser в `lib/citations.ts` фильтрует. **Новые файлы:** `web/lib/confidence.ts` — pure `computeConfidence(answer, sources): ConfidenceVerdict` (мемоизировано в page.tsx через `useMemo`); `web/components/chat/ConfidenceBadge.tsx` — bg + border + text одного цветового семейства (emerald/amber/orange/rose/destructive) с light/dark вариантами + цветной dot 8px + uppercase label + reason-строка. ARIA `role="status"` + `aria-label`. **Что НЕ сделано** (queued): info-tooltip объясняющий каждый tier (app-day-26+), ML-классификатор faithfulness (FactScore-style — после набора golden feedback'а), пороги per-style (concise может оправдать меньше citations — app-day-26+ после калибровки на реальных запросах). Концепт-док `docs/concepts/21-confidence-indicator.md`. **Tests:** `pnpm --filter web tsc/lint/build` clean.

- **rag-day-23:** Pāli-глоссарий и query-expansion. Перед encode'ом пользовательский запрос обогащается канонической палийской леммой плюс её EN/RU значениями из DPD — закрывает gap multilingual encoder'а на bare-Pāli (`jhāna`) и кириллических транслитерациях (`джхана`). Источник: [`sc-voice/ms-dpd`](https://github.com/sc-voice/ms-dpd) (CC-BY-NC 4.0, condensed JS-версия Digital Pāḷi Dictionary), 50 060 уникальных лемм с **100% покрытием EN и 99.99% RU** переводов. Скрипт `scripts/build_dpd_glossary.py` парсит 4 `.mjs`-файла ms-dpd в `data/glossary/dpd_full.json` (10 MB, коммит); raw `.mjs` (~15 MB) в `.gitignore`. Ручной кириллический слой `data/glossary/cyrillic.yaml` — 155 high-frequency терминов с 288 вариантами написаний (джхана/дхьяна, ниббана/нирвана, камма/карма, дхамма/дхарма…) собран по Ньянатилоке + theravada.ru. Phase 3 план — расширить до 500-2000 RU. Новый `src/processing/glossary.py::Glossary` + `load_glossary` + `expand_query` с алгоритмом: токенизация → lookup → top-2 EN + top-2 RU + дедуп против оригинальных токенов; `max_terms=5`, `max_meanings=2`. Hook-up: `RAGService` принимает `glossary` в конструкторе и вызывает `expand_query` до encode'а если `expand_pali=True`; `QueryRequest.expand_pali: bool | None` (None → server default), `PipelineMetadata.expand_pali` (effective, не запрошенное); `Settings.glossary_expand_pali_default: bool = False`. - **Mini-eval findings (`docs/EVAL_PALI_GLOSSARY.md`):** прогон на synthetic golden v0.0-extended (n=100). **Первая итерация без фильтра** дала смешанный сигнал: `ref_hit@1` +4 pp, но `ref_hit@5` −3 pp, fixed=2 / regressed=5. Failure-анализ выявил, что DPD содержит односимвольные служебные палийские слова (`a`, `na`, `va`) и обычные английские заимствования (`Buddha`, `sutta`), которые шумно матчили на английских запросах: расширение `Buddha` → «Awakened One Будда Пробужденный» в каждый запрос про Будду размывало semantic context. **Diacritic-guard эвристика** (`_has_pali_diacritics`): расширяем только токены с палийскими диакритиками (`jhāna`, `paṭiccasamuppāda`) или кириллицей (`джхана`); pure-ASCII пропускаем — encoder и так понимает английские заимствования в буддийском контексте. После фикса: `ref_hit@5` 0.450 → 0.450 (parity), `ref_hit@1` +1 pp, `ref_hit@20` +1 pp, fixed=1 / regressed=1, MRR +0.005. Шум устранён, но и большая часть upside — потому что в v0.0+ только 7/100 RU и 2/100 pli QA.
- **Targeted mini-eval (`docs/EVAL_PALI_GLOSSARY_TARGETED.md`):** Чтобы измерить пользу глоссария там где он *должен* работать, сделан отдельный набор `docs/eval/golden_pali_targeted.yaml` — 100 QA: 50 RU (cyrillic) + 30 bare-Pāli (с диакритиками) + 20 mixed (RU предложение + Pāli термин). Все 53 уникальных `expected_works` сверены с корпусом. Прогон с дефолтным `max_meanings=2` выявил **recall/precision tradeoff**: recall на bare-Pāli растёт **+13 pp на ref_hit@20** (0.400 → 0.533), но precision на top-1 проседает **−5 pp** (0.090 → 0.040), MRR −0.024. Природа эффекта — расширенный запрос (~10-20 лишних токенов) даёт encoder'у больше семантического сигнала (recall ↑) но теряет фокус (precision ↓).
